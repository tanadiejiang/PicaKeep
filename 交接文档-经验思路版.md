## §1 远程服务与连接池

> 精简版对应：[§1 远程服务与连接池](交接文档-精简最新版.md#1-远程服务与连接池)

### 现象

- 只要滚动经过"服务端实际不存在内容"的漫画/封面，之后**全部远程档**（已下载/图集/收藏/图片收藏）都无法载图。
- 离开页面、跨标签都恢复不了，**唯有热重启（hot restart）/重开 App 才好**。
- logcat 一片 `getUrl` 的 `TimeoutException 0:00:05`，**没有任何 404**（说明不是服务端拒绝，是客户端拿不到连接）。

### 根因与诊断路径

**决定性证据**：热重启重建整个 Dart isolate → `RemoteLibraryClient` 单例 + `HttpClient` 连接池全部清零 → 恢复。离开页面只销毁 widget，单例状态还在 → 不恢复。所以坏的是**单例持有的连接池**。

**两个泄漏源：**

1. **孤儿连接**（最关键）
   - `getUrl(...).timeout(5s)` / `openUrl(...).timeout(5s)` 超时时只丢弃前端 Future（`rethrow`），但底层那条"开连接"的 `Future<HttpClientRequest>` **还在跑**。
   - 它稍后成功返回一个 `HttpClientRequest`，但 catch 已退出、`request` 局部变量是 null → **没人 close/abort 它** → 这条连接被永久钉在 keep-alive 池里。
   - 每超时一次，可用连接 -1，累积到 `maxConnectionsPerHost`（原 12）就全局永久卡死。

2. **404 不 drain**
   - 非 2xx（如缺失封面 404）原来直接 `request.abort()` 就抛异常。但 `close()` 拿到 response 后 socket 归 response 所有，abort 在该状态下不可靠。
   - 每个缺失封面漏一条连接，滚过 >12 个（原池大小）就耗尽。

**触发场景**：滚动"没内容的漫画"→ 缺失图片让服务端响应慢 → 连接被多占几秒 → oversubscribe → 产生第一个孤儿 → 雪球。

**为什么退出页面再进还是卡**：`HttpClient` 在 `RemoteLibraryClient` 单例里，widget 销毁不影响它。

### 修法（全部已落地）

1. **孤儿回收 `_openWithTimeout(connecting, timeout)`**
   - 超时后给原 `connecting` future 挂 `.then()`，等它最终返回 request 时 `abort()`，把连接还回池子。
   - `loadImageWithProgress` 的 `getUrl` 和 `_sendRequest` 的 `openUrl` 都改走它。**这是关键修复。**

2. **404 drain 代替裸 abort**
   - 非 2xx 路径改为 `await response.drain<void>().timeout(5s)` 把响应体读空，连接干净归还；drain 自身超时才回退 abort。

3. **响应体保护 `_guardedImageBody`**
   - body 套 15s idle timeout（超时 abort + 抛错）。
   - `finally` 里对"未正常读完"（cancel/error）的情况 abort，正常读完则不动（保留 keep-alive 复用）。
   - 旧代码把裸 `response` 直接当 stream 返回——body 卡住或被丢弃时 socket 永远 pinned。

4. **并发闸 `_ImageConcurrencyLimiter`**（FIFO 信号量）
   - `loadImageWithProgress` 拿连接前先拿令牌。
   - 两个独立预算：**阅读闸** `settings[114]`（reader 大图，默认 6）/ **浏览闸** `settings[115]`（封面缩略图，默认 8）。
   - 设置入口：设置-阅读 / 设置-浏览，范围 1~12，实时生效。
   - 令牌释放：早退路径（404/超时/socket/其它异常）就地 `releasePermit()`；正常路径交给 `_guardedImageBody` 的 `finally`。

5. **池扩容**
   - `_httpClient.maxConnectionsPerHost` 从 12 提到 **24**（≥ 阅读闸 12 + 浏览闸 12 两上限之和）。
   - 保证正常并发下 `getUrl` 永不排队、永不超时、压根不产生孤儿。

6. **双池分流**
   - `_httpClient`（bulk，24 连接）：图片 bulk 池，给 reader 大图和封面。
   - `_controlClient`（control，6 连接）：控制面池，只给 JSON 请求用（封面已搬走）。
   - 封面必须走 bulk 池：封面网格和 JSON 列表/`/status` 同屏，封面占 control 池会把列表/探测饿死。

7. **`_trimToLimit` 节流**
   - 每张封面下载后不再立即全盘扫描。改为 30s 节流 + 不重入（`_trimInFlight` / `_lastTrimAt`）。

### ❌ 不要再走的路

- **不要**对 `getUrl()` / `openUrl()` 直接 `.timeout()` 就完事。**这是孤儿连接的来源**——超时只丢弃前端 future，底层请求还在跑。**必须用 `_openWithTimeout`**：超时后挂 `then` 回收。
- **不要**在 timeout 路径里只 catch 不 abort。Dart `HttpClient` 的 socket 不会自动释放。
- **不要**只给建连/收 header 包 `.timeout()` 就以为"请求有超时保护"。**响应体（body）流必须单独保护**：套 idle timeout，未读完时 `abort()`。裸 `await for(response)` 是这套 codebase 最隐蔽的池泄漏来源。
- **不要**让封面下载走 `_controlClient`（`lightweight: true`）。封面网格和 JSON 列表同屏，封面占 control 池会把列表饿死。
- **不要**新增 `HttpClient` 实例——应统一走 `_httpClient` / `_controlClient` 两个池。
- **不要**在"每张封面下载后"无脑跑全盘扫描类清理。网格批量加载必须节流 + 不重入。

## §3 加密压缩包

> 精简版对应：[§3 加密压缩包](交接文档-精简最新版.md#3-加密压缩包)

### 子问题 1：ZIP 内部章节名乱码（§28）

- **现象**：外层漫画标题正常，但 ZIP/CBZ 内部章节名/分组名乱码。
- **根因**：`ArchiveEntry.path` 来自 central directory 的原始 filename bytes。上游库可能已经错解过；`UTF-8 decode` 成功 ≠ 结果正确（例如 GBK 字节流被当 UTF-8 解不会抛异常但结果是乱码）。
- **修法**：
  - 若 ZIP 显式带 UTF-8 flag → UTF-8
  - 未标记时先保护纯 ASCII 路径
  - 对 `utf8 / shiftJis / gbk` 候选结果做启发式比较，不回退到"utf8 不抛异常就算对"
- **诊断原则**：外层标题正常、内部章节乱码 → 追 `central directory raw bytes → ArchiveEntry.path → archive chapter realNames → display helper` 这条链。不要先归因到 UI 拼接或缓存。

### 子问题 2：加密 ZIP 假解锁（§30）

- **现象**：1.7 GB 加密 ZIP 输任意非空密码都"解锁成功"，但封面破图、阅读器每页 `Invalid image data`。
- **根因**（三层，互相掩盖）：
  1. `tryUnlock` 只调 `backend.openIndex(...)` 以"能列目录"作为成功条件。ZIP 中央目录**从来不加密**、列目录不需要密码，所以任何密码都判对。
  2. `readEntry` 解密失败时 catch 落到 fallback，fallback 直接对加密字节做 `RawZLibFilter` inflate → 乱码字节喂给 `ImmutableBuffer` → `Invalid image data`。
  3. 中央目录解析不识别 WinZip AES extra field `0x9901`，加密 entry 的真实压缩方法从未被读出。
- **修法**：彻底绕开 archive 库的 `ZipDecoder`，自己实现加密 entry 解密链：
  - `_parseCentralDirectoryWithOffsets` 追加识别 `0x9901` → `isAesEncrypted` / `aesStrength` / 内层真实 `compressionMethod`
  - `readEntry` 主路径重写：加密 entry 走 `_readAesEntry` 或 `_readZipCryptoEntry`，非加密走 `_readEntryFallback`
  - `_readAesEntry`：读 local file header → 切 salt/pwdVerify/cipher/file mac → `_pbkdf2HmacSha1` 派生 key → 密码两字节校验失败抛 `wrongPassword` → HMAC 校验 file mac → `_aesCtrXorZip` XOR 解密 → 内层 inflate
  - `_readZipCryptoEntry`：3 个 32-bit 密钥用密码字节初始化，CRC32 每字节滚动，12 字节 header 校验
  - `_readEntryFallback` 对 `match.isEncrypted` 直接抛 `wrongPassword`，杜绝加密字节走裸 inflate
- **`tryUnlock` 真实验密码**：`openIndex` 后挑一个最小加密 entry 实际 `readEntry`，成功才 `setSessionPassword`。

### 子问题 3：AES-ZIP inflate 报错（§32）

- **现象**：修复假解锁后，AE-1 + AES-256 + deflate（如 `114514.zip`）报"压缩包已损坏"，而 AE-2 + ZipCrypto 包正常。
- **根因**：`RawZLibFilter.inflateFilter()` 默认 `windowBits=15` 期待 zlib header，但 ZIP entry 压缩数据是**裸 deflate 流**（无 header），对应 `windowBits=-15`。
- **修法**：两处 `inflateFilter()` 均改为 `windowBits: -15`：
  - `_inflateRaw`（AES / ZipCrypto 解密后调用）
  - `_readEntryFallback` 内 method=8 分支

### 与 archive 库的兼容性

- archive 3.6.1：`ZipFile` 构造硬调 UTF-8 解码 local file header filename → 日文 ZIP 抛 `FormatException`。
- archive 4.0.9：修了 filename 但 zip64 extra field 的 dataSize 未被消费 → 本仓库 zip64 包撞 `RangeError`。
- **两版库都不能透明使用**。当前方案：`pointycastle` 直接依赖 + 自实现解密链，源码不 `import package:archive`。

### ❌ 不要再走的路

- **不要**只列目录就判断"解锁成功"。ZIP 中央目录不加密。必须实际读一个加密 entry。
- **不要**对加密字节走裸 inflate。`_readEntryFallback` 入口遇到 `isEncrypted` 直接抛 `wrongPassword`。
- **不要**换 archive 库版本就以为能透明解决。3.6.1 和 4.0.9 各有不兼容问题。

### 涉及文件

- `lib/foundation/archive/backends/dart_zip_backend.dart` — 加密解密链、编码修复、inflate windowBits
- `lib/foundation/archive/archive_reading_service.dart` — `tryUnlock` 真实验密码
- `lib/foundation/archive/archive_models.dart` — `ArchiveEntry` 新增 `isAesEncrypted`
- `lib/foundation/local_library.dart` — 加密包封面刷新
- `lib/pages/download_page.dart` — "忘记密码"功能
- `lib/pages/settings/archive_settings_page.dart` — 密码管理 UI 重构
- `pubspec.yaml` — 引入 `pointycastle: ^3.9.1`

### 局域网发现边界

- `mDNS / Bonjour`：服务端启动后发布 `_picakeep._tcp.local`，客户端查询该服务类型（query 带 QU 位请求单播回复），拿到 host/port 后仍请求 `/status` 校验。**服务端收到查询后单播回复到源地址**（绕过 Windows 多网卡组播路由问题）；主动广播（初始/定时）仍发 `224.0.0.251` 组播。mDNS 结果按 `地址:端口` 去重；同设备不同端口视为不同服务，保留为独立候选。
- `网段扫描`：保留旧 `/status` 探测行为，按已配置地址端口和当前私网网段逐个探测；不删除、不弱化，只是不再冒充 mDNS。当前显式“网段扫描”和 mDNS 兜底扫描都会扫完当前候选目标集合，聚合多个有效 `/status` 服务；当前已连接地址仍优先探测，但命中后不能阻止后续地址继续扫描。
- `mDNS 失败自动网段扫描`：独立设置位，默认开启；仅当主挡位为 `mDNS / Bonjour` 且 mDNS 无结果或 socket 失败时，本次发现临时调用原 `scan()`。主挡位为 `网段扫描` 时只执行旧扫描逻辑。
- 服务信息页与 APP能力页共用 `ServiceDiscoveryStrategySettings`，两处显示和保存的是同一套策略。
- 服务信息页最终候选数为 1 时直接应用；候选数 2+ 时弹选择窗口。窗口展示来源模式、mDNS instance/host/TXT 服务名、地址端口、漫画数量、延迟和 `/status` 描述；如果 mDNS 兜底到了网段扫描，窗口顶部必须显示兜底提示。
- 多设备环境没有弹选择窗口时，优先检查最终候选是否被同 `地址:端口` 去重、是否只有一个服务通过 `/status`、或是否出现“扫描命中后提前停止”的回归；不要先改 mDNS 发布协议。
- Android 上 mDNS 需要 `CHANGE_WIFI_MULTICAST_STATE` 和 multicast lock；没有组播能力的路由器/访客网络应切回“网段扫描”。
- Android/Windows 的 UDP 5353 绑定不要启用 `reusePort`；Android 真机出现 `Dart Socket ERROR ... reusePort not supported on this platform` 时，说明已进入 mDNS 绑定路径，优先检查这里而不是回到旧网段扫描逻辑。

### 服务信息页跳动边界

- 服务信息页空闲跳动优先查页面自身：按钮里的隐藏 `CircularProgressIndicator` 即使透明也可能继续驱动 ticker，空闲态应只保留固定尺寸占位，不构建运行中的加载圈。
- `App.serviceStatsVersion` 会在服务端请求 begin/end 时变化。它只代表统计值变化，不应触发完整 `_reloadSnapshot()` 和 `_loading=true`，否则页面会在“在线”和“刷新中”之间来回闪。
- 统计通知走静默刷新：读取新快照、对比统计相关摘要，确有变化才更新 `_snapshot`；无变化不 `setState`。手动刷新、配置变化、运行态变化仍保留完整刷新和明确 loading 反馈。
- 这个问题与 mDNS / 网段扫描模式没有直接关系。发现按钮的加载反馈只跟 `_discovering` 相关，不应为了治理服务信息页跳动改发现策略。

### 架构约束

- **两套底层 store + 服务端聚合视图**：`LibraryTrashStore` + `LocalTrashStore`，兼容已有数据和 `download.db` 恢复能力。
- **WebSocket 接线**：前后台生命周期、指数退避重连、25s 心跳、30s 陈旧超时、`/status` 降级探测（8s 节流）。
- **阅读期延迟刷新**：收到变更先缓存 `pendingSignature`，退出阅读再统一刷新。

### ❌ 不要再走的路

- **不要**在服务端 `_notifyLibraryChanged()` 里调 `rescanResources()`。
- **不要**在服务端 favorites handler 里 `await init()`。
- **不要**把 `Uri.decodeComponent(segments[i])` 加回去。
- **不要**把 Windows mDNS 漏发现直接改成“用网段扫描代替 mDNS”。正确做法：保留真实 `mDNS / Bonjour`，排查顺序 ① `_sendResponse` 查询回复是否单播到源地址 ② 多网卡组播加入 ③ A 记录地址过滤 ④ UDP 5353 防火墙。
- **不要**在网段扫描命中当前已连接服务后 `break`。显式“网段扫描”需要聚合同一候选网段中的多个 `/status` 服务，当前已连接地址只能优先探测，不能终止后续扫描。

## §5 动态取色与设置交互

> 精简版对应：[§5 动态取色与设置交互](交接文档-精简最新版.md#5-动态取色与设置交互)

### 子问题 1：动态取色致 ANR（§36.4 / §34）

- **现象**：logcat `dynamicColors.start/done` 每 ~6s 刷一次，持续给主线程加压，偶发 `picakeep 没有响应`。
- **根因**：MIUI 每 ~6s 发一次 `onConfigurationChanged`。原生 `MainActivity.onConfigurationChanged` 节流只有 1000ms，6s > 1s 每次都穿过 → 转发 `dynamic_theme` `changed` → Dart `getCorePalette()` + 双 `toColorScheme()` + `setState` 全量重建。
- **修法**：`MainActivity.onConfigurationChanged` 改为**仅当 uiMode 夜间标志真正翻转才转发**。壁纸调色板不走这个回调，首次取色由 initState 独立触发。删掉失效的 `DYNAMIC_THEME_NOTIFY_MIN_INTERVAL_MS` 与 `lastDynamicThemeNotifyAt`。
- **验证**：正常使用时控制台不再每 6s 刷 `dynamicColors`，只在手动切系统亮/暗时刷一次。

### 子问题 2：设置滑块不跟手（§36.4）

- **现象**：设置里滑块圆点不跟手，要等别处触发整页 rebuild 才跳到新值。
- **根因**：`buildExploreSettings` 无状态顶层函数 + `ReadingSettings` 是 `StatelessWidget`，内联 `Slider.onChanged` 只写 `appdata.settings` 不 `setState`。
- **修法**：三个滑块各封装成小 StatefulWidget：
  - `_ComicTileSizeSlider`（浏览-漫画块大小）
  - `_ImageFavoriteSizeSlider`（浏览-图片收藏大小）
  - `_TapTurnRangeSlider`（阅读-点按翻页识别范围）

### 涉及文件

- `android/app/src/main/kotlin/lingxue/picakeep/MainActivity.kt` — onConfigurationChanged 节流
- `lib/pages/settings/explore_settings.dart` — 滑块 StatefulWidget 封装
- `lib/pages/settings/reading_settings.dart` — 同上

## §6 Android 启动与系统 UI

> 精简版对应：[§6 Android 启动与系统 UI](交接文档-精简最新版.md#6-android-启动与系统-ui)

### 子问题 1：阅读器顶部黑条（§33.1）

- **现象**：从特定入口（如 LauncherBlue alias）启动应用后，阅读器顶部有黑条。
- **根因**：logcat 出现 `D/VRI[LauncherBlue]` — launcher 入口是 `activity-alias` 而非 `MainActivity` 本身。Flutter embedding v2 靠 `<meta-data android:name="io.flutter.embedding.android.NormalTheme">` 切换 NormalTheme，但这只挂在 `<activity>` 上，没复制到各 alias。alias 启动时窗口停留在 LaunchTheme，无 `windowLayoutInDisplayCutoutMode=shortEdges` 也无 `statusBarColor=transparent`。
- **修法**：给全部 13 个 `activity-alias` 补上：
  ```xml
  android:theme="@style/LaunchTheme"
  ...
  <meta-data
    android:name="io.flutter.embedding.android.NormalTheme"
    android:resource="@style/NormalTheme"/>
  ```
- **⚠️ 后续新增 alias 必须同样带上这两项。**

### 子问题 2：状态栏动效（§33.2）

- **现象**：隐藏顶栏时状态栏瞬间消失（无淡出动画）。
- **根因**：V1.8 把 `_hideReaderSystemUi` 从 `SystemUiMode.immersive` 改成了 `immersiveSticky`。前者走 Android 标准淡出动画（~300ms），后者 hard-switch 瞬间消失。
- **修法**：改回 `SystemUiMode.immersive`。
- **副作用**：`immersive` 下用户从屏幕顶端下拉时状态栏会停留几秒才自动收回（`immersiveSticky` 会在失去交互后立即藏回）。为动效体验接受此副作用。

### 子问题 3：启动页相关（§34.2）

- **深色模式跟随应用内设置**：Android 12+ 启动画面由 `system_server` 在进程启动前绘制。改用 `UiModeManager.setApplicationNightMode()`（在 `onCreate` 里 `super.onCreate` 之后调用）。删除原 `attachBaseContext` 的 `createConfigurationContext + applyOverrideConfiguration`（会在 Activity 不重建时残留锁定）。
- **双桌面图标修复**：删除 `MainActivity` 的 `MAIN/LAUNCHER` intent-filter，入口全交给 `<activity-alias>`。曾尝试 debug overlay 但导致 debug 包双图标，已删除。
- **全 vector 启动图标**：用 potrace 矢量化 → `launch_icon_full.xml` + `launch_splash_icon.xml`（0.6 缩放避系统裁剪）。

## §7 本地回收站与删除链路

> 精简版对应：[§7 本地回收站与删除链路](交接文档-精简最新版.md#7-本地回收站与删除链路)

### 核心链路

```
删除 → TrashManager 移入回收站 → 真删源 download.db 行 + 保存 sourceDbPath/sourceDbId/sourceDbRowId/sourceDbTimeMillis
恢复 → 优先按原 rowid 插回 download.db（被占用则普通插入保留原 time）
彻底删除 → 物理删除文件 + 清理回收站记录
```

### 关键修复

| 问题 | 根因 | 修法 |
|------|------|------|
| 删除后列表残留 | 移入回收站时未删源 download.db 行 | 移入时真删，保存来源信息供恢复 |
| 恢复后只剩一条 | 受限 download.db 读不到被误判"不存在"→ 误建空库覆盖 | 先区分"不可读"和"真的不存在" |
| 跨应用 managed 资源删除抛 PathNotFoundException | `DownloadManager.init()`/`createSync()` 误触他人沙箱包根 | 识别"他人应用沙箱"路径 → 强制走 Shizuku/Root 通道 |
| 删除时长时间转圈 | 预先递归统计整个目录大小 | 改为优先复用现有条目大小；引入 pending 中间状态 |
| 删除后 reload 失败弹窗 | reload 异常被当删除失败 | reload 失败只记日志，不覆盖删除结果 |
| 受限目录权限不足静默失败 | 旧逻辑允许"静默移入回收站/从列表移除" | 权限不足直接提示，不假装成功 |
| Shizuku UserService 偶发 DeadObjectException | 大 download.db 读取超时/断开 | 当前回退/跳过，不阻塞删除成功路径 |
| 图集回收站封面残留 | 移入时未清理封面缓存 | 记录封面元数据，恢复后修正 |

### Shizuku 本地已下载补充（§12）

- **现象**：Shizuku 下本地已下载扫描返回 0、或"我"页下载数量统计失败。
- **根因**：多个辅助入口在 Shizuku + 受限目录时仍偷偷调 `DownloadManager().init()` → 对受限路径做 `Directory.create()`/`sqlite3.open()` → `PathNotFoundException`。
- **修法**：新增 `LocalLibraryManager.shouldBypassDirectDownloadManagerForCurrentDownloads()` → 在 `managedDataSourceModeCurrentOnly` 且命中 Shizuku fallback 时统一绕开 `DownloadManager`，改走 `LocalLibraryManager` 受限目录读取链路。

### ❌ 不要再走的路

- **不要**在受限目录场景调 `DownloadManager().init()`。
- **不要**把 `dart:io` 读不到受限 `download.db` 误判成"源库不存在"然后建新空库。
- **不要**在删除前对整个目录做递归体积统计。
- **不要**对受限目录静默"移入回收站/从列表移除"假装成功。

- `lib/foundation/trash.dart` — 删除/恢复/彻底删除统一入口
- `lib/foundation/local_trash_store.dart` — 回收站持久化
- `lib/foundation/local_library.dart` — 扫描、Shizuku 绕行
- `lib/foundation/download.dart` — DownloadManager
- `lib/pages/trash_page.dart` — 回收站 UI、进度遮罩

## §8 已下载页三档隔离

> 精简版对应：[§8 已下载页三档隔离](交接文档-精简最新版.md#8-已下载页三档隔离)

### 三档架构

```
已下载页
├── 本地档：DownloadManager 本地扫描 → 8s 超时
├── 聚合档：本地 + 远程合并（单侧失败不拖垮另一侧）
└── 远程·已下载：远程 HTTP 快照 → 10s 超时
```

### 关键修复

| 问题 | 根因 | 修法 |
|------|------|------|
| 远程已下载被本地扫描失败拖住 | 三档共用加载入口，远程前先执行 `DownloadManager().init()` | 拆分为独立分支 |
| 一直转圈不超时 | 无页面级超时 | 本地 8s、远程 10s 超时 → 可重试空态 |
| 远程已下载把整根目录识别为一个"多章节大项" | 扫描逻辑错误 | 优先扫描子目录漫画列表 |
| Root/Shizuku 列表慢 | 首屏同步全量扫描 + 封面解析 | 改用 `download.db` 快速构建，封面/索引后台低速补齐 |
| 远程封面回滑变慢/消失 | provider key 不稳定（StreamImageProvider vs FileImage 换 key） | 统一为 `_RemoteLibraryCoverImageProvider` + `cacheWidth` 优化 |
| 已下载详情弹层标签多时挤压按钮 | 固定高度 | 改为自动高度 + 可滚动 |

### 封面缓存稳定性（§21）

- **根因**：第一次走 `StreamImageProvider`，磁盘缓存存在后切 `FileImage` → Flutter 图片缓存 key 不同 → 回滑时即使文件已落盘也不命中同一份内存缓存。
- **修法**：统一为稳定 key 的 `_RemoteLibraryCoverImageProvider`，优先读磁盘缓存；已下载页远程封面预热前 12 个；移动端 `cacheExtent` 提升到 1.0 屏。

## §13 网页后台与浏览器应用

> 精简版对应：[网页后台 / 浏览器应用](交接文档-精简最新版.md#网页后台--浏览器应用)

### 当前架构

网页后台现在不再是旧的单 Dart 字符串 `/admin` 页面，而是 `assets/web_console/` 下的静态 SPA：

```
assets/web_console/
├── index.html
├── css/common.css / admin.css / app.css
└── js/common.js / auth.js / shell.js / admin.js / app.js
```

服务端通过 `lib/server/web_console/web_asset_loader.dart` 与 `web_console_handler.dart` 从 `rootBundle` 读取这些资源；`pubspec.yaml` 已注册 `assets/web_console/`、`assets/web_console/css/`、`assets/web_console/js/`。旧的 Dart 字符串控制台 `lib/server/admin_web.dart` 已迁移删除，不再作为页面来源。`/admin` 兼容重定向，非 API/非静态路径走 SPA fallback，因此：

- `/`：应用面板
- `/app`：应用面板
- `/admin-view`：管理面板
- `/admin`：旧入口兼容，最终进入管理面板或重定向

### 计划一地基要点

- `server_app.dart` 先匹配所有 `/api/*`、`/status`、`/api/events`，最后才交给 `web_console_handler` 做静态资源 / SPA fallback。
- 所有 API 响应统一补 CORS；`OPTIONS` 直接返回 204，暴露 `Content-Range / Accept-Ranges / Content-Length` 供图片 Range 使用。
- `/status` 保持免登录，便于客户端探活；`/api/library/*` 是 Flutter 客户端远程资源公共接口，必须免网页 token；`/api/events` 免登录用于资源热同步；网页静态资源和登录接口免登录；`/api/admin/*`、`/api/console/*`、`/api/remote/proxy` 走 token 鉴权。
- `/api/remote/proxy?target=...&path=...` 是浏览器远程访问另一台 PicaKeep 服务端的唯一通道：只允许 GET，转发 Range/Accept，不透传 Cookie/Authorization，失败归一为 502/JSON error。
- `common.js` 的 `PicaKeepConsole` 是计划二/三/四/五/六共用契约，后续前端只应通过它访问 API、远程代理、token 图片 URL、source 与 remoteTarget。
- `admin.js` 最初由旧 `/admin` 平移而来；计划三再扩展管理 tab。不要再把业务逻辑塞回 Dart 字符串 HTML。

### 关键能力清单

1. **真实登录与网页专用账号**
   - `POST /api/console/login` 支持 `{username,password}`，空 username 兼容旧 admin 登录。
   - `GET /api/console/me` 回填当前用户；admin 可管理用户。
   - token 绑定 `userId + passwordHash`，改密码后旧 token 自动失效。
   - 账号系统只属于网页后台；Flutter 客户端没有账号功能。

2. **per-user 网页阅读历史**
   - 独立 sqlite：`web_console.db`，不复用客户端 `history.db`。
   - 只隔离历史；漫画库、收藏夹、图片收藏、回收站仍是服务端共享数据。
   - `GET/POST/DELETE /api/library/history` 给网页阅读器写回和 dashboard 历史条使用。

3. **应用面板**
   - 支持本地/远程/聚合数据源，封面墙 → 详情 → 阅读器。
   - 阅读器支持连续纵向滚动与翻页两种模式，模式持久化。
   - 图片 URL 必须通过 adapter：本地用 `api.imageUrl(relPath)`，远程用 `remote.imageUrl(target, relPath)`；不能直接把后端相对路径塞给 `<img>`。
   - 阅读器翻页/滚动会节流写回网页历史，dashboard 历史条可续读到上次 ep/page。

4. **dashboard 化与 app「我」页风格**
   - 顶栏删除“我全都要”，只保留“应用面板 / 管理面板”，并支持折叠抽屉。
   - dashboard 改为 app「我」页风格：历史卡、已下载、图集、资源库、图片收藏、工具胶囊。
   - 卡片视觉为描边透明底、成对网格、统一 chevron 与横向历史条。

5. **已下载 / 图集分流规则**
   - 网页不需要新增服务端字段；`/api/library/items` payload 已含 `rootId`。
   - 分流硬规则：
     ```text
     item.rootId.startsWith('custom_') == true  → 图集 / album
     item.rootId.startsWith('custom_') == false → 已下载 / 漫画 / comic
     ```

### 阅读器 App 化与侧栏多轮收口（后续多轮已执行）

在地基之后，又执行了多轮把网页 reader/侧栏向 Flutter 客户端对齐的计划（`plan/后端网页/` 下的二次修复、App 化复刻、侧栏回归/四次补齐、UI 微调一二三、两轮验收补修）。核心经验：

- **reader 空白 ≠ 图片加载失败**：DevTools 实测时 21 张图片已进入 DOM、网络图片 200、`naturalWidth/naturalHeight` 正常，但整页空白。根因是 shell/grid 下 `#console-root` / `[data-reader-root]` / `.reader-strip` / `.reader-img` 被压成 0 宽。排查口诀：**同时看 DOM 图片数、network 200、naturalSize、父级 `getBoundingClientRect().width`**，宽度为 0 才是真凶。
- **reader-active 布局不变量**：进入阅读器后主内容必须全宽，或把 `#console-root` 明确放到正确 grid column；左侧栏 / 旧 `#topbar` 迁移时不能让内容列塌陷成 0 宽。
- **本地 reader 验证先切「本地」源**：数据源停在「远程」时会触发 `/api/remote/proxy?...target=http://127.0.0.1:9527...` 远程自代理，返回 502/401 干扰本地阅读验证。
- **阅读手势对齐 `lib/pages/reader/touch_control.dart`**：用 pointerdown/move/up 状态机，不要退化成单个 `pointerup`。
  - 中心点击切换上下栏；仅当工具栏隐藏且命中边缘区才翻页。
  - 连续滚动模式边缘判定必须用**视口高度 + `event.clientY`**，不能用整条连续 reader 的 `getBoundingClientRect().height`（高达数万 px，会把屏幕中心误判成顶部翻页区，导致点击永远翻页、永不展开上下栏）。
  - 指针释放语义要区分：正常释放（`lostpointercapture` / document `pointerup` / `blur`）只清 longpress/zoom transient，**不要清掉刚安排的单击 tapTimer**；只有 `pointercancel` / `visibilitychange(hidden)` / 长按触发才可清 tapTimer。否则单击切栏会被吞。
  - 双击/长按是 viewer 级缩放（`.reader-zoom-layer` 包裹 `.reader-strip`，对 layer 做 transform，避免单页 `overflow:hidden` 裁切），缩放 origin 固定到点击点；放大后可随指针拖动平移（`state.reader.zoom.panX/panY` + CSS 变量 `--reader-zoom-pan-x/y`，按缩放层尺寸 clamp 边界）；松手/取消/blur/visibilitychange 必定还原。
- **侧栏对齐 `download_page.dart` + `side_bar.dart`**：桌面右侧贴边 App 面板（hero 封面+标题 → meta chips → tags → 章节 → sticky footer 双按钮），移动端底部 sheet；章节区双列（`.app-panel-episode-list` 两列 grid，窄屏回退单列），单章不被 grid 拉伸成超高空框。
- **上下栏功能对齐 `lib/pages/reader/tool_bar.dart`**：顶栏=返回+标题+阅读设置；底栏=上一章/滑条/下一章 + 全屏/收藏/自动翻页/章节/保存（去掉分享）；页码在左下角。
- **多轮 UI 微调还涉及**：dashboard 卡片、grid 详细/列表视图、详情页推荐区（新增 `/api/library/items/{id}/recommendations`，算法在 `server_app.dart`）、管理面板浅色 token 化。

### 网页 reader 历史保存 / 滚动恢复 / 历史页对齐已下载（后续已执行）

这一轮把网页阅读历史的「写」与「读」两端、以及历史独立页向「已下载」对齐。改动**只在 `assets/web_console/js/app.js` 与 `css/app.css`**，未动客户端/服务端。

- **ESC 退出不保存进度（只有返回键保存）**：ESC（`onGlobalKeydown`）与顶栏返回按钮其实**同走一个 `back()`**，并不是两条分支。差异在阅读翻页/滚动会用 `scheduleReaderHistoryWrite()` 挂一个 **1500ms 防抖**定时器再落盘；退出时 `cleanupReader()` 旧逻辑**直接 `clearTimeout` 把待写定时器清掉、不补写**。返回键之所以"看起来正常"，只是因为唤出顶栏+移鼠标+点击通常已超过 1500ms、防抖早已落盘；ESC 可在窗口内瞬退，于是最后页码被取消。修法：`cleanupReader()` 改调 `flushReaderHistory()`——清定时器**后**按当前实时页（滚动模式用 `currentScrollPage()` 回写 `pageIndex`）调一次 `writeReaderHistoryNow()`；`lastHistoryKey` 去重保证未移动时是 no-op。ESC / 返回键 / 任何跳转都统一经 `cleanupReader`，一处补写全修好。
- **从历史打开总停在第 1 页（仅滚动连续模式）**：页码数据全程正确（`openHistory` 传 `pageIndex: record.page` → `renderReader` 存入 → `renderReaderContent` clamp），分页模式直接以该 index 渲染所以正常。滚动连续模式旧实现只有**一次** `requestAnimationFrame(() => scrollToPage(pageIndex))`，而 `scrollToPage` 用 `scrollIntoView`：目标页之上的图片是懒加载，跳转那刻上方各页高度未稳定，落点几乎在顶部；紧接着 `pageObserver` 按当前可视页**把 `state.reader.pageIndex` 覆写**（并触发历史写入），停回第 1 页。修法：`restoreScrollPosition(target)`——置 `state.reader.restoring=true`，多帧（rAF）重试 `scrollToPage(target)` 直到 `currentScrollPage()===target` 稳定或超时（约 25 帧兜底），`restoreRaf` 在 `cleanupReader` 取消；`pageObserver` 回调加 `if (state.reader.restoring) return;` 守卫，恢复期间不让观察器覆写 pageIndex。`target<=0` 直接不重试。
- **历史卡第二行「0」+ 历史页对齐已下载**：服务端历史记录（`server_app.dart` 的 `upsertHistory` / `web_user_store.dart` 的 `toJson`）**只持久化 `target/title/cover/ep/page/maxPage/readEpisode`，没有 author/subtitle/tags**。`readEpisode` 是 int 集合，回传 `[0]`；旧 `normalizeHistoryPayload` 用 `String(firstValue(['readEpisode',…]))` 把它变成字符串 `"0"`，卡片 `item.readEpisode || item.subtitle` 就渲染成"0"。修法：① `normalizeHistoryPayload` 不再用 `readEpisode` 当展示文案；② 作者/副标题与标签改由 `renderHistoryPage` 用 `settle()` 并行拉 `history.list` + `fetchItemsForSource(当前源)`，建 `Map(id → {subtitle, tags})` 对每条历史补全（资源库缺该条目就留空、隐藏该行）；③ 启用历史页搜索——`updateToolbar` 把 `historyPage` 加进 `searchInput.disabled` 白名单与 `viewSwitchEl.hidden` 白名单，**复用同一个 `state.grid.query` 与 `state.grid.viewMode`（与已下载共用记忆）**，search input / clear / view-switch 的 handler 各加 `historyPage` 分支调 `renderHistoryPageContent`，过滤用现成 `filterItems`；④ 卡片改用 `.app-card`(`historyPosterCardHtml`) / `.app-detailed-card`(`historyDetailedCardHtml`) 结构，保留右上角 `.app-history-remove`。
- **详细视图标题溢出覆盖下一行**：`.app-detailed-card` 用 `align-items: stretch`，使 `.app-detailed-title` 被拉伸、`-webkit-line-clamp` 的 `-webkit-box` 高度不收敛，标题换行到 3+ 行盖住下面卡片。修法：`.app-detailed-title` 补全 `display:-webkit-box; -webkit-box-orient:vertical; overflow:hidden; -webkit-line-clamp:3; white-space:normal`，并加 `.app-history-entry .app-detailed-title { align-self: start }` 让 clamp 真正生效（不被 stretch 撑开）。标签经库 join 后显示：详细视图前 3 个（`.app-detailed-tags`）、海报视图前 2 个（`.app-detailed-tags.app-history-tags`），复用 grid 既有标签样式。

### 网页应用面板新增收藏夹入口 + 服务端收藏封面 fallback + 卡片对齐客户端（后续已执行）

这一轮做了四件事：① 网页 dashboard 新增「收藏夹」入口（接 `/api/library/favorites`）；② 服务端收藏夹封面在本地 `coverPath` 失效时按 candidates 匹配资源库兜底；③ 通用详细档卡片向客户端对齐、标签全显示且跟随主题色；④ 收藏夹文件夹列表与 dashboard 滚动收口。改动落在 `assets/web_console/js/app.js`、`css/app.css`、`lib/server/server_app.dart`、`js/admin.js`。

- **收藏项封面破图 + 卡片不可点（核心 bug）**：服务端 `_buildFavoriteItemPayload` 原本只回 `name/author/type/tags/target/time/coverUrl`，**没有 `itemId`**；而客户端/网页用 `__matched: !!itemId` 判断是否匹配到资源库，于是 `__matched` **永远 false** → 封面被强制清空、卡片标为 `aria-disabled` 不可点。根因不是 cover 路由本身（`/api/library/favorites/{folder}/{target}/cover` 一直可用），而是 payload 缺字段。修法：服务端 GET 列文件夹内容时对每个收藏项跑一次 `_resolveFavoriteResourceItem(snapshot, item)`，匹配到就把 `itemId/id/displayId/sourceDisplayName/imageCount/totalBytes/updatedAt` 一并写进 payload，`__matched` 才能正确为 true。
- **收藏封面 fallback（coverPath 失效但资源库有内容）**：cover 路由改为三级兜底——① 本地 `item.coverPath` 文件存在就直接 `_fileResponse`（注意要看返回是否 404，不能只看路径非空）；② 命中 `_favoriteCoverFallbackCache`（key=`folder|type|target`）且文件在就用缓存；③ 都没有则 `_currentSnapshot()` + `_resolveFavoriteResourceItem` 按 candidates 匹配资源库项，取 `matched.coverPath`，为空再 `_scanner.resolveCoverPathOnly(matched, rootPath)`，成功后写入 fallback 缓存。`_recomputeLibrarySignature` 里 signature 变化时**连 `_favoriteCoverFallbackCache` 一起清**，避免资源移动后命中 stale 路径。
- **candidates 复用客户端逻辑**：`_buildFavoriteServerCandidates(item)` 直接调 `item.candidateDownloadIds()`（FavoriteItem 自带，含 ehentai/jm/hitomi/Ht/nhentai 等前缀变体），再补 `target` 本身；`_findResourceItemByCandidates` 把每个资源项的 `id/title/displayId/sourceTitle/sourceDisplayName/subtitle/path` 收成 pool 做包含匹配。**不要**在服务端另写一套候选规则，会和客户端漂移。
- **cover 路由要带 `?type=`**：同一 target 在不同收藏类型下可能并存，`coverUrl` 末尾拼 `?type=${item.type.key}`，`_findFavoriteItemByTarget` 增加可选 `type` 参数按类型精确定位；DELETE 单项同样带 type。
- **`_rootPathForRootId` 从 config 解析**：`current_download`/`original_download` 取 `config.currentDownloadRoot`/`originalDownloadRoot`，`custom_{n}` 按下标取 `config.customLibraryRoots[n]`，越界返回 null。
- **详细档标题严格 2 行**：`.app-detailed-title` 同时写 `text-overflow:ellipsis` 和 `-webkit-line-clamp:2` + `white-space:normal` 会冲突，导致第三行漏出一点。修法：去掉 `text-overflow`，保 `display:-webkit-box; -webkit-box-orient:vertical; -webkit-line-clamp:2; overflow:hidden`，再补 `max-height: calc(1.35em * var(--app-detailed-title-lines))`（lines 默认 2）兜底像素漏出。历史页 `.app-history-entry .app-detailed-title` 仍是 3 行覆盖（§13 前一轮），本轮不破坏。
- **详细档标签全显示 + 跟随主题色**：`gridDetailedCardHtml` 去掉 `tags.slice(0,3)` 改全量渲染（`.app-detailed-tags` 本就 `flex-wrap:wrap` 自动换行）；标签底色**不写死**，用 accent 主题变量 `background: rgba(var(--accent-rgb),0.12)` + `color: var(--accent-2)` + `border: 1px solid rgba(var(--accent-rgb),0.28)`，圆角 8px，用户在设置里换 12 色主题时标签随之变色（等效客户端 `secondaryContainer` 的主题感）。

## §16 远程访问图集压缩包（含加密包）

> 精简版对应：[远程压缩包](交接文档-精简最新版.md#网页后台--浏览器应用)。本轮在 V1.9.15 之后执行。

### 核心设计：服务端把压缩包伪装成普通 item

客户端本地早已能读图集里的 ZIP/CBZ（`archive://` URI + `ArchiveReadingService`），但**服务端原本完全不扫压缩包**，客户端远程/网页都看不到。本轮让**服务端把每个压缩包扫成一个普通 item/episode**，页 URL 仍是 `/api/library/items/{id}/images/{ep}/{page}`，只是该路由命中压缩包时返回 archive 内字节而非磁盘文件。**客户端远程数据源消费 HTTP URL 的逻辑零改动**，只有加密解锁需要增量。

### 关键实现点

- **共享分组逻辑抽出**：`lib/foundation/archive/archive_episode_builder.dart`（新）顶层 `buildArchiveEpisodes(ArchiveIndex)` / `pickArchiveCoverEntry`，从 `local_library.dart` 私有静态方法提取，客户端与服务端复用，排序结果须完全一致（页码顺序对用户可见）。
- **服务端扫描**：`local_resource_scanner.dart` `_scanCustomRootItems`（图集 custom_ 根）末尾加 `_collectArchiveFiles` + `_scanCustomArchiveItem`：`getIndex` 解析中央目录（不需密码）→ `buildArchiveEpisodes` 分章 → `imagePaths` **直接存 `archive://?path=&entry=` URI**（类型仍是 `List<String>`）。`ServerResourceItemSummary`/`Episode` 新增 `isArchive/archiveEncrypted/archivePasswordMatched/archiveFormat/archivePath`，`toJson` 与 `_buildLibraryItemPayload` 透传 `itemKind`(archive/directory) 等。
- **取图/封面路由**：`server_app.dart` 图片路由命中 `isArchiveUri(imagePaths[page])` → `_archiveBytesResponse`（新，`readEntryBytesByUri` 取字节 + Content-Type by entryPath，加密读失败抛 → 403 `archive locked`/404）；封面路由对 `item.isArchive` 同样走 `_archiveBytesResponse`。
- **加密 unlock 接口**：`POST /api/library/items/{id}/archive/unlock {password}` → `ArchiveReadingService.tryUnlock`（进程单例会话密码，成功后取图/封面自动用上）→ 成功清 `_deepItemCache`/`_coverPathCache` + `rescanResources()`。另有 `GET .../archive/status`。密码只走 body（`logRequests` 不打 body）。
- **客户端远程解锁**：`RemoteLibraryComicItem.fromJson` 解析 archive 字段 + `needsArchivePassword` getter；`RemoteLibraryClient.unlockArchive`；`archive_password_dialog.dart` 抽 `onVerify` 回调（本地传 `tryUnlock`、远程传 `unlockArchive`）；解锁入口补到 `local_library_page._handleItemTap`、`download_page`（列表 onTap + 上下文菜单 + 详情 read）、`local_comic_detail_page`。
- **网页前端**（`app.js`）：`ensureArchiveUnlocked(item)` 在进 reader/详情前判断 `archiveEncrypted && !archivePasswordMatched` 弹密码框调 unlock；normalize 透传 `itemKind/isArchive/archiveEncrypted`；远程经 `consoleApi.remote.post(.../archive/unlock)`、本地经 `consoleApi.api.post`。

### 踩坑（务必记住）

- **headless 服务端没注册 archive backend**：`ArchiveRegistry.initDefaults()` 原仅在 `lib/main.dart` 调，`bin/main_server.dart`/`PicaKeepAdminServer.start()` 没调，且 `register()` 无去重。修：`initDefaults` 改幂等（`if (_backends.isEmpty)`）+ `start()` 早期调一次。否则独立服务端进程 `backendForOrThrow` 抛 `unsupportedFormat`。
- **`_ensureDeepItem` 对 custom_ 直接返回 shallow**：压缩包 item 的 episodes/imagePaths **必须在浅扫阶段建全**（`openIndex` 只读中央目录、廉价，可行）。
- **`_fileResponse` 只能服务磁盘真实文件**（`File(path).exists()`），archive entry 不是文件，必须用 `_archiveBytesResponse` 返回字节。
- **加密包封面**未解锁抽不出字节 → 路由返回 403/404 → 端侧显示锁占位。
- **会话密码进程内存、重启即失**：服务端重启后远程取图返回 403，端侧需重新弹解锁框。

### ❌ 不要再走的路

- **不要**给服务端 archive 另写一套分组/封面逻辑——用共享的 `buildArchiveEpisodes`/`pickArchiveCoverEntry`，否则与客户端漂移。
- **不要**让 unlock 密码进 URL/query 或写进日志 body。
- **不要**以为 `_fileResponse` 能服务 archive entry。

### 涉及文件
- `lib/foundation/archive/archive_episode_builder.dart`（新）、`lib/foundation/local_library.dart`（改调共享）
- `lib/server/local_resource_scanner.dart`、`lib/server/server_app.dart`、`lib/foundation/archive/archive_registry.dart`
- `lib/foundation/remote_library_data_source.dart`、`lib/components/archive_password_dialog.dart`、`lib/pages/local_library_page.dart`、`lib/pages/download_page.dart`、`lib/pages/local_comic_detail_page.dart`
- `assets/web_console/js/app.js`、`lib/server/web_console/remote_proxy_handler.dart`（代理支持 POST 以转发 unlock）

## §17 阅读器「图片最大显示宽度」滑块在连续模式失效与闪烁

> 客户端 `settings[43]` 开关 + `settings[116]` 数值（默认 980，范围 600–1600）。本轮多处修复。

- **翻页模式（mode 1/2/3、5/6）未应用限宽**：`image_view.dart` 的 `buildType123`/`buildType56` 用 `PhotoViewGallery` 直接吃满视口，没读 `settings[116]`。修：在 gallery **容器外层**包 `Center + SizedBox(width: _clampReaderImageWidth(constraints.maxWidth))`。
- **连续模式（mode 4）完全不变**：`ScrollablePositionedList` 是 sliver，**给 item 的横向约束是 tight = viewport 全宽**，item 内 `ComicImage` 的 `SizedBox(width)` 被拉伸/忽略。修：在 `ScrollablePositionedList` **外层**限宽居中，而非依赖 item 内部 width。
- **「只全屏能调、只调一点」**：外层 `PhotoView.customChild` 的 child 写死 `MediaQuery.size.width`（全屏宽，非阅读区可用宽），且 item 内又做了一次 `_clampReaderImageWidth` + `height/width<1.2` 兜底二次收缩。修：限宽基准改用 `LayoutBuilder` 真实 `constraints.maxWidth`，**只限一次（外层）**，item 内不再重复 clamp。
- **拖动滑块时图片闪烁**：`_createReaderImageRequest` 连续模式 `cacheWidth=(layoutWidth*dpr).round()` → `ResizeImage`，拖动时 layoutWidth 每帧变 → cacheWidth 每帧变 → **provider key 每帧变 → 反复重解码**。修：**解码宽度与显示宽度解耦**——`layoutWidth`（决定 cacheWidth）用阅读区全宽（稳定），`ComicImage.width` 才用实时 listWidth（纯布局缩放，不重解码）。

### ❌ 不要
- **不要**靠 sliver item 内部 `SizedBox(width)` 限宽——被 tight 约束吃掉，要在列表外层限。
- **不要**用 `MediaQuery.size.width` 当限宽基准——窗口模式 ≠ 阅读区宽，要用 `LayoutBuilder` 约束。
- **不要**让 `cacheWidth` 跟随滑块每帧变——会重解码闪烁，解码宽度固定、显示宽度才实时。

### 涉及文件
- `lib/pages/reader/image_view.dart`（`buildType4`/`buildType123`/`buildType56`/`_createReaderImageRequest`、helper `_isReaderImageWidthLimited`/`_clampReaderImageWidth`）

## §20 网页后台滚动/布局修复（侧栏切换 / 滚动条 / toolbar 间距 / reader 穿透）

> 精简版对应：[网页滚动/布局收口](交接文档-精简最新版.md#网页后台--浏览器应用)。

### 侧栏展开/收起触发整页滚动

- **根因**：`#console-shell` 用 `min-height:100vh`（由内容撑高）+ `transition: grid-template-columns`，侧栏切换改列宽 → 主内容区重排高度微增 → 整页可滚。刷新重新布局又消失。
- **修**：`#console-shell` 改 `height:100dvh; overflow:hidden`（固定视口高、永不整页滚）；`#console-root` 改 `height:100%; overflow-y:auto`（内容超高时在内部滚动）。切换侧栏只改列宽、不再影响外层高度。

### 主页无用滚动条

- `#console-root overflow-y:auto` 在内容刚好时仍显滚动条 → `scrollbar-width:none` + `::-webkit-scrollbar{display:none}` 隐藏。
- dashboard toolbar（本地/聚合/远程）与历史卡间距过近 —— `:has()` 规则未命中，改用给 toolbar 补 `padding-bottom` 或 dashboard 首卡加 margin-top 的方式。

### reader 滚动穿透

- 非全屏时 `.app-reader` 是 `overflow:hidden`，长内容走 document 滚 → 背后 grid 列表跟滚。非全屏也加 `overflow-y:auto` + 外层 document 锁滚。

### ❌ 不要
- **不要**在 `#console-shell` 上再用 `min-height:100vh`（应固定 height）。
- **不要**依赖脆弱的 `:has(> .app-main-toolbar + .app-view > .app-dashboard)` 兄弟链——用更直接的选择器。

### 涉及文件
- `assets/web_console/css/common.css`（`#console-shell`/`#console-root`/`html.pk-reader-active`）
- `assets/web_console/css/app.css`（`.app-reader`/`.app-main-toolbar`）

### 当前最重要的后续方向：两状态高度学习

移动浏览器实际可以按两个稳定视口高度处理：

- **工具栏展开**：地址栏/底部栏可见，视口高度最小，需要底部垫高。
- **工具栏收起**：地址栏/底部栏隐藏或缩起，视口高度最大，不需要浏览器 chrome 垫高。

推荐算法：

1. 仅在移动端、非 PWA standalone 下启用。
2. 当前高度优先取 `visualViewport.height`，取不到再用 `window.innerHeight`。
3. 首次打开默认认为浏览器工具栏展开：记录为 `minViewportHeight`，并使用默认垫高。
4. 后续只要观察到更大的高度，更新 `maxViewportHeight`，视为工具栏收起状态。
5. 一旦 `maxViewportHeight - minViewportHeight` 足够大（例如超过 24px），就用当前高度接近哪一端来决定 pad：
   - 接近最小高度：`--mobile-browser-chrome-pad = 默认垫高`。
   - 接近最大高度：`--mobile-browser-chrome-pad = 0px`。
   - 中间态可用阈值切换，不必复杂线性插值。
6. PWA standalone 始终 `0px`。
7. CSS 侧只能有一个主要消费点，避免 double cushion。

### ❌ 不要

- **不要**继续叠多个底部补偿：`.app-panel-content` 和 `.app-panel-footer` 同时吃 `--mobile-browser-chrome-pad` 会产生大空白。
- **不要**把 footer 固定/绝对定位作为默认方向；之前三段式 sheet 已试过，交互不稳定且用户不认可。
- **不要**再加 touch/window scroll booster 试图强制隐藏浏览器工具栏；普通网页没有可靠强制能力。
- **不要**只看 `visualViewport` 的实时差值就认为是真实底部遮挡；不同浏览器/ROM 表现不一致，优先按“最小高度/最大高度”状态机处理。
- **不要**在未重新读取当前 `app.css` / `shell.js` 的情况下继续改底部垫高；用户中途可能手动回滚过。

## §25 Linux 桌面构建 — sqlite3 下载卡死

> 精简版对应：[Linux 桌面构建](交接文档-精简最新版.md#linux-桌面构建)

### 现象

- 远端 `flutter build linux --debug` 一直停在 `Building Linux application...`，无后续输出。
- 检查远端进程：cmake 连接 `194.195.208.62:443`（sqlite.org）处于 ESTABLISHED 但数据不流动（`wchan: do_sys_poll`）。
- `ps aux` 可看到 `cmake -P .../download-sqlite3-populate.cmake` 进程卡住。
- cmake configure 报错：`Cannot find source file: .../sqlite3.c`。
- 本地 tar → pscp → 远端解包后每次都复现，因为 `build/` 不在 tar 里、`_deps/sqlite3-src/` 被清空。

### 根因

`sqlite3_flutter_libs` 插件的 `linux/CMakeLists.txt` 使用 `FetchContent_Declare(sqlite3 URL https://sqlite.org/2026/sqlite-autoconf-3520000.tar.gz)` 从 sqlite.org 下载源码。`sqlite.org` 在中国大陆不可达（被墙），导致 cmake configure 阶段的 `FetchContent_MakeAvailable` 卡在 TCP 连接超时。

### 修复方案（已落地）

**捆绑 sqlite3 合并源码到项目 + CACHE 变量跳过 FetchContent**

1. **生成合并源码**：从 GitHub 下载 sqlite 源码（可直连）→ `./configure && make sqlite3.c` 生成 `sqlite3.c`（~9.4MB）和 `sqlite3.h`（~680KB）。
2. **放入项目**：存放在 `linux/sqlite3/sqlite3.c` 和 `linux/sqlite3/sqlite3.h`。
3. **修改 `linux/CMakeLists.txt`**：在 `include(flutter/generated_plugins.cmake)` 之前添加：

```cmake
# Pre-bundled sqlite3 amalgamation to avoid downloading from sqlite.org
set(sqlite3_SOURCE_DIR "${CMAKE_CURRENT_SOURCE_DIR}/sqlite3" CACHE PATH
  "Pre-bundled sqlite3 amalgamation source directory")
set(sqlite3_POPULATED TRUE CACHE INTERNAL
  "Skip FetchContent download for sqlite3")
```

**原理**：`FetchContent_MakeAvailable` 内部检查 `sqlite3_POPULATED` 变量——如果为 TRUE，直接跳过下载。`sqlite3_SOURCE_DIR` 被设为 CACHE 变量，指向捆绑的合并源码目录。插件 CMakeLists.txt 中的 `add_library(... "${sqlite3_SOURCE_DIR}/sqlite3.c")` 会直接使用本地文件。

### 涉及文件

| 文件 | 角色 |
|------|------|
| `linux/sqlite3/sqlite3.c` | sqlite3 合并源码（~9.4MB，随项目提交） |
| `linux/sqlite3/sqlite3.h` | sqlite3 合并头文件（~680KB，随项目提交） |
| `linux/CMakeLists.txt` | 添加两行 CACHE 变量（在 `include(generated_plugins.cmake)` 之前） |
| 远端 `~/.pub-cache/hosted/pub.dev/sqlite3_flutter_libs-0.5.42/linux/CMakeLists.txt` | 插件原始 CMakeLists.txt（无需修改，CACHE 变量覆盖其 FetchContent） |

### 更新 sqlite3 版本的标准操作

当 `pubspec.lock` 中 `sqlite3_flutter_libs` 或 `sqlite3` 包版本升级时：

1. 确认新版本对应的 sqlite3 上游版本号（如 `3.52.0` → tag `version-3.52.0`）。
2. 下载源码：`curl -L -o /tmp/sqlite3.tar.gz 'https://github.com/sqlite/sqlite/archive/refs/tags/version-X.XX.X.tar.gz'`
3. 解压并生成合并文件：
   ```bash
   tar xzf /tmp/sqlite3.tar.gz --strip-components=1
   ./configure && make sqlite3.c
   ```
4. 替换 `linux/sqlite3/sqlite3.c` 和 `linux/sqlite3/sqlite3.h`。
5. 确认 `linux/CMakeLists.txt` 中的变量设置仍然正确。
6. 如果插件 CMakeLists.txt 中的 `FetchContent_Declare` URL 版本号也变了，CACHE 变量机制依然生效——只要 `sqlite3_POPULATED=TRUE`，FetchContent 就不会触发下载。

### 远程构建流程适配

远端构建流程（tar → pscp → rm -rf 解包 → flutter build）已无缝支持：
- `linux/sqlite3/` 目录不在 tar 的 `--exclude` 列表中，会随源码一起传到远端。
- 远端无需额外操作，cmake configure 自动使用捆绑的 sqlite3.c。

### ❌ 不要再走的路

- **不要在远端 pub cache 里改 CMakeLists.txt**：`flutter pub get` 可能覆盖，且无法随项目源码分发。
- **不要用 `file://` URL 指向远端固定路径**：`/home/azusa/.sqlite3/` 这类路径绑定特定机器，开源后别人没有。
- **不要只清理 lockfile 而不处理 FetchContent**：lockfile 只是 Flutter 工具锁，真正卡住的是 cmake 的 `ExternalProject/FetchContent` 下载。
- **不要尝试从 `sqlite.org` 直连下载**：该域名在中国大陆不可达，`ghproxy.net` 等代理也不稳定。
- **GitHub 源码 ≠ autoconf 包**：GitHub 下载的是开发树（无 `sqlite3.c`），必须运行 `./configure && make sqlite3.c` 生成合并文件后才能用。

### 诊断步骤（再次遇到时）

```bash
# 1. 检查是否有卡住的 cmake 下载进程
ps aux | grep -E 'cmake|sqlite' | grep -v grep

# 2. 检查是否在连 sqlite.org
ss -tnp | grep -E 'cmake|194.195.208'

# 3. 确认捆绑文件存在
ls -la linux/sqlite3/sqlite3.c linux/sqlite3/sqlite3.h

# 4. 确认 CMakeLists.txt 有 CACHE 变量
grep -n 'sqlite3_SOURCE_DIR\|sqlite3_POPULATED' linux/CMakeLists.txt
```

### 26-1 详情页推荐全量重算（确定根因，已修）

- **现象**：进入/退出本地漫画详情页、下滑切 AppBar 标题，profile 实测 build 帧 180~196ms。
- **根因**：`local_comic_detail_page.dart` 每次 `build()` 都同步调 `_buildRecommendations()`，
  对全库 `_localItems`（可能几百上千项）逐项跑正则相似度（`_nameTopics`/`_normalizeName`/
  `_bigrams`/`_nameSimilarity`）+ 整体 sort。任意 setState（滚动切标题/翻页/dispose 前）都重算一遍。
- **修法**：结果缓存成字段 `_recommendationsCache` / `_recCacheComic` / `_recCacheMode`。
  原方法拆成 `_computeRecommendations(mode)`，缓存命中（`identical(_recCacheComic,_comic)` +
  同模式 + 未失效）直接返回。`_loadLocalItems` 的 setState 里显式 `_recommendationsCache=null`
  失效（因 `_localItems` 是 final 列表原地 mutate，identical 检测不到内容变化）。
- **诚实边界**：缓存只在「同一详情页实例」内生效。**每个详情页首次进入仍会算一次**（193ms），
  这是缓存的固有首次成本，不是 bug。下滑/翻页的重复 180ms 已消除（profile 复测降到 ~27ms）。
  要再降首次只能优化算法本身或挪 isolate，本轮未做。
- **顺带修**：推荐项点击从裸 `MaterialPageRoute` 改 `App.pushInner`（详见返回栈 Bug 文档）。

### 26-2 工具页预返回动画 saveLayer（推断性修复，已确认有效）

- **现象**：工具页预返回（predictive back）缩放动画，120Hz 下 raster 卡顿，trace `saveLayer×52`。
- **修法**：`tools.dart` 的 body 外包一层 `RepaintBoundary`，把整页栅格化成单层，
  返回缩放只合成一层纹理，避免每帧重做几十个卡片的离屏 saveLayer。
- **真机复核结论**：返回动画变顺，说明 saveLayer 来源确实在工具页本身（多张
  `Card.outlined(Clip.antiAlias)` + `AnimatedCrossFade`），推断成立，保留。
  （若当时无改善，分支方案是给被揭开的下层页面封面 tile 加 RepaintBoundary。）

### 26-3 已下载页转圈变长 = 远程探测阻塞本地加载（logcat 实锤，已修）

- **现象**：进「已下载」转圈 3.5s 才出列表（本地 4158 部）。用户反馈「修复后才变长」，
  但实为巧合——这次远程服务恰好离线，触发了一直存在的串行超时。
- **logcat 埋点实锤**（`[DLPROBE]` 分段计时）：
  ```
  checkRemoteAvailability done +3004ms  remoteAvailable=false   ← 3s 全耗在这（超时）
  _loadComics done          +3466ms  count=4158               ← 本地加载只 462ms
  ```
- **根因**：`download_page.dart` 的 `reload()` 里 `remoteAvailable = await
  _checkRemoteAvailability()` **串行挡在** `_loadComics()` 前。远程离线时它走到 ~3s 超时。
  而 local 视图的 `_loadComics` 分支**根本不读 `remoteAvailable`**（只 aggregate/remote 视图读）。
- **修法**：local 视图直接 `remoteAvailable=false` 跳过探测，只 aggregate/remote 视图才 await。
  `showSourceSelector` 用 `_hasConfiguredRemoteServer` 兜底，跳过探测不影响视图选择器显示。
- **⚠️ 与踩坑第 1 条的区别**：附录踩坑说「不要给 `_checkRemoteAvailability` 加缓存掩盖问题」。
  本修法**不是加缓存**，是 local 视图压根不需要这个探测结果，所以不调用——没有掩盖任何远程问题，
  远程视图该探测照样探测。方向一致：不靠缓存粉饰，而是消除不必要的等待。

### 26-4 已下载/图集 loading 全屏遮罩 → 列表区转圈（已改）

- **现象**：切远程档时整页被居中转圈替换，AppBar/视图选择器都消失。
- **原则**（用户定）：任何档位加载都只在列表区转圈，加载完刷新视图显示，不整页覆盖。
- **修法**：`download_page.dart` 的 `build` 去掉 `logic.loading ? 整页转圈Scaffold : 正常`
  三元，始终渲染正常骨架；`_buildComics` 在 `comics.isEmpty && loading` 时返回
  `SliverFillRemaining` 居中转圈，否则才空状态。`SmoothCustomScrollView` 就是标准
  `CustomScrollView`，支持 `SliverFillRemaining`。

### 26-5 图集远程档滚动卡顿 = GC 风暴（实测定位，两段叠加修复）

- **现象**：滚远程图集列表，120Hz 下 jank 15~25%，但 **build/raster 全低（<4ms），
  elapsed 全靠 vsyncOverhead 撑高**（最重帧 vsync 100~300ms）。用户体感：
  「加载完封面不卡，卡在还在加载+显示封面时」。
- **排除法**（`[ALBUMPROBE]` 埋点，`local_library_page.dart`）：每秒 listItemBuild、
  `_coverImageProvider`、`_coverFile` 累计耗时——**全部 <1ms/秒**。Dart 侧 widget 构建
  与封面 provider 路径完全无辜（`BaseImageProvider` 的 `==`/`hashCode` 基于 key 正确，
  缓存命中正常）。
- **真凶（trace 关键字）**：`CollectOldGeneration` / `Scavenge` / `StartConcurrentMark` 大量出现。
  **老年代 GC 暂停**就是 vsync 100~300ms 尖峰的来源。滚动时大量封面字节在堆上快速分配/驱逐，
  撑爆新生代、晋升老年代，触发昂贵 old-gen GC，主线程冻结。
- **修法（两段，都在 image_loader 基建层，默认不影响其他子类）**：
  1. **限尺寸解码**：`base_image_provider.dart` 加 `int? get targetDecodeWidth`（默认 null）。
     子类返回宽度时，`decode` 走 `getTargetSize` 按宽缩放解码。
     `_RemoteLibraryCoverImageProvider` override `=> 480`（封面显示约 150~200px，高 DPI 留余量）。
     远程封面原图可能数千 px，全尺寸解码是解码耗时+内存的浪费。
  2. **绕过堆字节缓存**：`base_image_provider.dart` 加 `bool get cacheRawBytes`（默认 true，
     保持现有行为）。`_RemoteLibraryCoverImageProvider` override `=> false`——它已有磁盘缓存
     （`_RemoteLibraryCoverDiskCache`），原始压缩字节不必再进那个 50MB 静态堆 `_cache`，
     堆字节驻留正是老年代 GC 的压力源。
- **profile 实测（06 改后 vs 05 仅限尺寸，均 120Hz 可比）**：
  ```
  maxVsync   303ms → 51ms   ← 最严重尖峰降到 1/6（体感最大改善：不再几百ms冻顿）
  putIfAbsent 195  → 33     ← 绕堆缓存后图片缓存进出大减
  NewGC      153  → 85      ← 新生代 GC 近半
  ```
- **诚实边界**：OldGC 次数没明显减少（解码后的 image 对象仍晋升），但**单次代价大降**
  （sumVsync 也降）。剩余 ~13% jank 集中在「封面解码上屏瞬间」，是远程图片列表固有成本，
  120Hz 放大感知，60Hz 下更轻。继续挖（限并发解码/预解码）收益递减，本轮到此为止。

### 26-6 工具-缓存大小限制不生效（够不着 + 不触发，已修）

- **现象**：工具页显示「684MB / 524MB」，缓存超限却不降。
- **根因（读代码确认，双重失效）**：`_RemoteLibraryCoverDiskCache._trimToLimitInner`
  （`remote_library_data_source.dart`）——
  1. **范围错**：`totalBytes` 统计 `App.cachePath` + `dataPath/cache` 全部（684MB ✓），
     但可删文件池 `collectFilesUnder` **只收 `remote_library_covers` 子目录**。
     占大头的 `App.cachePath` 图片缓存算进总量却**删不到**。
  2. **时机窄**：trim 只在**远程封面下载后**触发（`ensureDownloaded` 末尾），平时不跑。
- **修法（保持原 LRU 设计：删最旧的到刚好低于限制，不清空）**：
  1. 可删文件池扩到 `App.cachePath` + `dataPath/cache` **全部**（与统计/`eraseCache` 一致），
     全局按修改时间 LRU，保护 `protectedPath`。
  2. 加公开入口 `RemoteLibraryDataSource.trimCacheToLimit()` → `_RemoteLibraryCoverDiskCache.
     trimToLimitNow()`（忽略节流、`_trimInFlight` 串行保护、无 protectedPath）。
  3. `tools.dart` 的 `_refreshCacheSize()` 统计前先 trim；`_editCacheLimit` 改完限制后调
     `_refreshCacheSize`，新限制立即生效。
- **缓存语义安全**：`App.cachePath`（`getApplicationCacheDirectory`）和 `dataPath/cache` 都是
  可被系统随时清理的缓存目录，`eraseCache` 本就全删，LRU 删除无副作用。

### 26-7 profile APK 体积翻倍 = 增量打包空洞（非代码问题）

- **现象**：反复构建后 `app-profile.apk` 从 ~45MB 涨到 93.6MB。
- **解剖**（扫本地文件头）：真实压缩内容仅 43MB，但有 **52MB 对齐填充**——845 个空名、
  comp=0、extra≈64KB 的孤儿条目（不在中央目录里，`zipinfo` 只认 1187 个真实文件）。
- **根因**：AGP 8.12 zipflinger **增量打包**，反复 build 删改条目留下的空隙不压实，
  累积成空洞。**非代码/资源/配置问题**。
- **修法**：`flutter clean` 全量重打包即压实（93.6MB → 41.1MB）。无需改任何 gradle 配置。
- **⚠️ 教训**：清产物用 `flutter clean`，**不要手删 `build/app/intermediates/` 半截目录**——
  会破坏 Gradle 增量状态一致性，导致「找不到 launch activity」假错误。

## §27 本地已下载列表滚动卡顿

> 精简版对应：[§ 本地已下载性能与图集封面特权修复](交接文档-精简最新版.md#本地已下载性能与图集封面特权修复第三轮rootshizuku-覆盖)
> 数据来源：`Z_性能数据/01~06`，60/120Hz profile 包 + DevTools 逐帧

### 27-1 常驻后台封面缓加载（已删）

- **现象**：root 模式本地「已下载」滚动卡顿，退出页面/应用后仍触发 root 权限提示。
- **根因**：`_warmManagedDownloadCache` 挂在 `LocalLibraryManager` 单例上，通过 `_backgroundCacheTask` 链式 fire-and-forget，遍历全部 items 逐项走 root 特权通道迁移封面，不绑任何页面生命周期。
- **修法**：删 `_warmManagedDownloadCache` 及调用点、`_backgroundCacheTask` 字段、`_resolveCoverPathForCacheOnly`、`_isCoverCachePaused`/`pauseBackgroundCoverCache`/`resumeBackgroundCoverCache` 死代码。封面缓存完全惰性化。

### 27-2 按需迁移避开活跃滚动

- **现象**：删后台任务后，滚动中按需迁移仍在帧间隙打 root 通道（vsync 尖峰 67ms）。
- **修法**：`_drainCoverResolveQueue` 循环开头加 `if (_isScrollInteracting) break;`（在 `removeFirst()` 之前），活跃滚动只入队不打通道；`ScrollEndNotification`→140ms idle→重启 drain 补齐。闭环：滚动流畅→停下补封面。

### 27-3 本地封面降采样解码

- **现象**：折返滚动 / 周期性卡顿，封面全分辨率解码（800×1200→150px 格子），大位图快速填满 image cache 触发老年代 GC。
- **修法**：`_DownloadedPageComicTile` 开 `super.optimizeCoverDecode = true`，按布局宽×dpr×1.35 做 `ResizeImage` 解码期降采样，不压缩源文件。

### 27-4 无封面项负缓存

- **现象**：无封面 managed 项每次折返走 ~11 次特权通道重探（cover.* exists + listDir 多轮）。
- **修法**：
  - `_ensureManagedDownloadCoverCache`：解析失败持久化 `noCoverSentinel` 到 source cache JSON，下次直接返回 null。
  - `resolveCoverPathForItem`：哨兵直接短路，不走 fallback。
  - `_coverResolveFailedIds`（页面级 Set）：同会话折返拦截。
  - `reload()` 调 `invalidateNoCoverSentinels()` 清除标记（用户主动刷新时重新探测）。
- **诚实边界**：无封面项本就只能显示占位图标，持久化标记让它跨会话都不再触发 root 通道。

### 27-5 trustStorageFromDatabase 假阳性

- **现象**：滑到 db 有记录但目录已删除的项时卡顿。
- **根因**：root 模式下 `trustStorageFromDatabase=true` 把 db-only 项一律标为 `localStorageExists=true`，入封面队列后逐个走特权通道探测不存在目录。
- **修法**：两处 `? true` 改为 `? _managedDownloadDirectoryExistsInIndex(...)`（查内存 Set，零 I/O）。`sourceDirectoryNames` 在加载开头已一次性列出，纯内存比对。

### 涉及文件（§27）

- `lib/foundation/local_library.dart`（27-1 删后台缓存、27-4 哨兵+refresh、27-5 trustStorage）
- `lib/pages/download_page.dart`（27-2 滚动暂停 drain、27-3 optimizeCoverDecode、27-4 页面负缓存）

## §28 root/shizuku 下图集封面破图

> 精简版对应：[§ 本地已下载性能与图集封面特权修复](交接文档-精简最新版.md#本地已下载性能与图集封面特权修复第三轮rootshizuku-覆盖)
> 共 10 轮递进修复，核心根因链条：解析层→渲染层→provider 内短路，三层全堵死了 root/shizuku 的外部存储读取

### 28-1~28-7 核心根因与最终修复

**为什么拖了 10 轮**：

1. **解析层**（第 1-2 轮）：`resolveLocalComicCoverPath` 同步路径无目录扫描兜底；`PrivilegedStorageAccess` de:io `existsSync=true` 但 `listSync` 受 scoped storage 静默空→特权通道永远不触发。
2. **渲染层**（第 3-7 轮）：图集封面全用裸 `FileImage`/`Image.file`，不经过特权通道 → 解析层算出了路径但渲染层读不出字节。
3. **provider 内短路**（第 8 轮，真正开关）：`imageProviderForLocalPath` 自身 `existsSync()` 对 root/shizuku 外部路径恒 true→裸 FileImage→`StreamImageProvider` 特权回退从不触发。这就是「没能用上权限」。

**最终修法**：

- `PrivilegedStorageAccess.listDirectoryEntries`/`readFileBytes`：dart:io 空结果时强制回退特权通道。
- 图集封面渲染统一改为 `imageProviderForLocalPath`（privileged-aware），去所有裸 `FileImage`/`Image.file`。
- `imageProviderForLocalPath` 特权模式跳过 `existsSync` 快路径，直走 `StreamImageProvider`+`_readFileBytes`（先 dart:io，空时回退特权通道）。
- 非特权（full-access/桌面/沙箱）：existsSync 可靠→FileImage 快路径，零回归。

### 28-8 外层 source 卡片拼贴 + 惰性 provider

- 外层 source 卡片加 `_LocalRootCollage` 2×2 子项封面拼贴。
- `coverImageProviderForItem` 惰性 provider：封面由 provider 自身惰性拉取，不依赖外部 setState 时序，消除退出再进封面闪空。

### 28-9 底栏/侧栏面板封面

- `DownloadedComicInfoView._buildCover` 本地项统一走 privileged-aware provider，去裸 `Image.file`。

### 28-10 撤销权限闪退定性

- 撤销 MANAGE_EXTERNAL_STORAGE 时 Android 强制 kill 应用进程是 OS 级固有行为，非 bug 不修。

### 涉及文件（§28）

- `lib/foundation/local_library.dart`（回写 localCoverPath、resolveChildCoverPath、惰性 provider、imageProviderForLocalPath 特权短路）
- `lib/foundation/privileged_storage_access_flutter.dart`（空回退）
- `lib/pages/local_library_page.dart`（_LocalRootCollage、_coverImageProvider 惰性）
- `lib/pages/local_comic_detail_page.dart`（异步兜底、去 existsSync、惰性回退）
- `lib/pages/download_page.dart`（drain 回写统一、底栏面板 provider）

### ❌ 不要再走的路

- **不要**在 root/shizuku 下用裸 `File(path).existsSync()` 判断外部路径封面是否存在——挂载点可见但读不了，恒 true。
- **不要**用裸 `FileImage`/`Image.file` 渲染图集封面——必须走 `imageProviderForLocalPath`。
- **不要**给 `imageProviderForLocalPath` 加 existsSync 快路径拦截——特权模式下它就是堵死特权通道的短路器。
- **不要**把「侧栏/底栏能显示封面」当成特权通道可用的信号——阅读器侧栏（eps_view）根本不渲染封面图，只显示章节文字列表。

## §30 开屏剪贴板大文本 ANR 修复

> 精简版对应：[§ 交互优化与开屏剪贴板 ANR 修复](交接文档-精简最新版.md#交互优化与开屏剪贴板-anr-修复)

### 现象

冷启动后「我」页面图集数量迟迟不刷新，真机 profile 定位到一次 **6.4 秒主线程冻结**。

### 根因（决定性证据）

MIUI logcat：`APP_SCOUT_HANG duration=5006ms` → `PerfMonitor longMsg wall=6389ms`。
Java 栈卡在 `JSONStringer.string → Formatter.format`——Flutter `Clipboard.getData` 的返回值
整段塞进 `JSONMethodCodec` success envelope，在 Java 主线程逐字符转义 JSON 字符串。
剪贴板有超大文本（截图自动写入或长文复制）时编码就要数秒。

触发时机：`main_page.dart` 启动 +2.8s `_scheduleClipboardCheck → checkLocalClipboard → Clipboard.getData`。
期间所有 `setState`（含 me 页面计数刷新）都被主线程冻结堵住。

> Dart-timeline perfetto trace 是盲区：卡顿在原生 Java 线程（JSON 编码），Flutter trace 不给它打 slice。

### 修法（必须走原生通道，纯 Dart 改不了）

`Clipboard.getData` 无参数限制读取长度。绕开内置通道：

- **Android 原生**（`MainActivity.kt`）：新增 `lingxue.picakeep/clipboard` MethodChannel，
  method `getText(maxChars)`。`primaryClipDescription` 非 `text/*` 时直接返回 null
  （MIME 短路跳过截图等 `image/*` 内容）；否则 `coerceToText` 后 `substring(0, maxChars)`。
- **Dart 端**（`local_app_links.dart`）：`_readClipboardTextTruncated(maxChars: 10000)` 调
  原生通道；`MissingPluginException` 时回退 `Clipboard.getData` 并在 Dart 侧截断。

### ❌ 不要再走的路

- **不要**用 Flutter 内置 `Clipboard.getData` 在启动路径上读剪贴板——JSONMethodCodec 编码大文本会冻主线程。
- **不要**在 Dart 侧尝试截断——`Clipboard.getData` 无参数，整段文本已在原生侧编码完毕。


以下来自 `交接文档-临时-远程收藏与图片收藏.md` §5，逐条不删：

1. **不要**给 `_checkRemoteAvailability` 加缓存"掩盖"问题。它失败的真正原因 99% 是连接池被拖垮，不是探测本身慢。
2. **不要**把 `Uri.decodeComponent(segments[i])` 加回去。`pathSegments` 已自动解码。
3. **不要**在服务端 favorites handler 里 `await init()`。
4. **不要**用 `App.globalTo` 打开阅读器。永远用 `App.pushInner`。
5. **不要**用 `App.globalContext` 调 `ScaffoldMessenger.of`。用页面/tile 自己的 BuildContext。
6. **不要**把图片收藏远程档点击候选缩成只有 `image.id`，要带 `title` + 前缀变体 fallback。
7. **不要**在 timeout 路径里只 catch 不 abort。Dart `HttpClient` 的 socket 不会自动释放。
8. **不要**只给建连/收 header 包 `.timeout()` 就以为"请求有超时保护"。**响应体（body）流必须单独保护**：套 idle timeout，且在被取消/出错（未读完）时 `abort()`。裸 `await for(response)` 是这套 codebase 最隐蔽的池泄漏来源。
9. **不要**新增 `HttpClient` 实例（应通过 `_httpClient`/`_controlClient` 这两个），否则又开始单池竞争。
10. **不要**在"每张封面下载后"无脑跑全盘扫描类清理（`_trimToLimit`）。网格批量加载会把它放大成几十次扫盘，必须节流 + 不重入。
11. **不要**对 `getUrl()` / `openUrl()` 直接 `.timeout()` 就完事。**这是孤儿连接的来源**：超时只丢弃前端 future，底层开连接的请求还在跑，稍后成功返回的 `HttpClientRequest` 无人 abort，永久钉住一条连接；累积到 `maxConnectionsPerHost` 就全局假死，**只有热重启/重开 App 能恢复**（因为连接池在单例里）。**必须用 `_openWithTimeout`**：超时后给原 future 挂 `then`，等它最终返回 request 时 `abort()` 回收。判断是不是这个坑的铁证——logcat 全是 `getUrl` 的 5s TimeoutException、没有 404、且热重启后立刻恢复。
12. **不要**让 `maxConnectionsPerHost` 小于"所有并发闸上限之和"。否则正常并发就会 oversubscribe → `getUrl` 排队 → 超时 →（若没孤儿回收）雪球。当前 `_httpClient=24` ≥ 阅读闸 12 + 浏览闸 12。改并发闸上限时记得同步检查这里。
13. **不要**忘了释放并发令牌。`loadImageWithProgress` 拿令牌后：早退路径（404/超时/socket/其它异常）就地 `releasePermit()`；正常路径把释放责任交给 `_guardedImageBody` 的 `finally`（正常读完/出错/被取消都会释放）。漏一条路径就会泄漏槽位，最终把闸门耗尽变成假死。
