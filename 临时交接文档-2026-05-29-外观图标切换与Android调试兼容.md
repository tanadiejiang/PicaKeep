### 3.1 `flutter run` 找不到包名 / 启动 Activity 的兼容修复

- `android/app/src/main/AndroidManifest.xml`

- 当前项目 Android 启动图标切换方案使用的是 `activity-alias`
- Flutter 3.41.6 的 `flutter run` 在解析 Android 启动入口时，只扫描主 `AndroidManifest.xml` 里的 `<activity>`，不会识别 `<activity-alias>`
- 因此原先即使 Android 系统本身能运行，`flutter run` 也会报：
  - `package identifier or launch activity not found`

最终有效修复：

- `android/app/src/main/AndroidManifest.xml`
  - 顶层 `<manifest>` 已补 `package="lingxue.picakeep"`
  - `MainActivity` 已补一个 `MAIN + LAUNCHER` 的 `intent-filter`，仅用于让 Flutter 工具识别调试入口
- 新增 `android/app/src/release/AndroidManifest.xml`
  - 用 `tools:node="removeAll"` 移除 `MainActivity` 上的 `intent-filter`
  - 目的是避免 release 构建中把调试兜底入口带进正式包，继续让正式包以 `activity-alias` 图标方案为主

结论：

- 当前源码状态下：
  - `flutter run` 调试可识别启动入口
  - release 构建仍尽量不干扰原有动态图标 alias 方案

### 3.2 Windows 多盘符 worktree 下 Kotlin 增量缓存崩溃修复

修改文件：

- `android/gradle.properties`

效果：

- 当前 worktree 下 `flutter run` 已可继续完成 Android 构建与安装

## 7. Android 启动页（SplashScreen）全面重做

### 7.1 背景与问题

本次会话在图标切换功能完成后，继续收口了 Android 启动页的三个问题：

1. **深色模式跟随应用内设置不生效**：切换"启用/禁用"后重启 App，启动页背景色不跟着变，只有系统深色开关有效。
2. **启动图标模糊**：原 `launch_icon_blue.webp`（192×192）被拉伸到 ~768px 显示，严重糊化。
3. **双桌面图标**：`MainActivity` 自身带 `MAIN/LAUNCHER` intent-filter，同时 `LauncherBlue` alias 也带，桌面出现两个图标。
4. **深色模式切回"跟随系统"后不生效**：之前用 `attachBaseContext` 强制锁定 uiMode，切回跟随系统时 Activity 不重建，锁定状态残留。

### 7.2 深色模式跟随应用内设置

**根因**：Android 12+ 的启动画面由 `system_server` 在 App 进程启动之前绘制，`attachBaseContext` 里的任何操作都无法影响已经画好的启动画面。

**修复方案**：使用 `UiModeManager.setApplicationNightMode()`（API 31+）。调用后系统会记住 per-app 夜间模式偏好，下次冷启动时 `system_server` 用该偏好解析启动主题资源。

**涉及改动**：

`android/app/src/main/kotlin/lingxue/picakeep/MainActivity.kt`：
- 删除 `attachBaseContext` 中的 `createConfigurationContext + applyOverrideConfiguration` 逻辑（这套方案会在 Activity 不重建时残留锁定状态，导致切回"跟随系统"后不生效）
- `onCreate` 里 `super.onCreate` 之后调用 `applyApplicationNightMode(readDarkModeSetting(this))`
- 新增 `applyApplicationNightMode(darkMode: String)` 私有方法：`"1"→MODE_NIGHT_NO`、`"2"→MODE_NIGHT_YES`、`"0"→MODE_NIGHT_AUTO`，API < 31 时 no-op
- 新增 `lingxue.picakeep/night_mode` MethodChannel，方法 `setMode`，供 Flutter 端在用户改设置时立即同步

`lib/tools/night_mode_channel.dart`（新增）：
- `NightModeChannel.instance.setMode(value)` 封装，非 Android 平台 no-op，异常静默吞掉

`lib/pages/settings/settings_page.dart`：
- 深色模式 `SelectSetting.onChanged` 里加 `unawaited(NightModeChannel.instance.setMode(value))`

**时序说明**：`setApplicationNightMode` 写完后，当前这次启动画面已经画完，本次冷启动看不到效果。需要：改完设置 → 退出 App → 再冷启动一次，第二次才生效。这是 Android 系统级限制，无解。

### 7.3 双桌面图标修复

**根因**：`MainActivity` 自身带 `MAIN/LAUNCHER` intent-filter（为 `flutter run` 调试兼容加的），同时 `LauncherBlue` alias 默认 `enabled="true"` 也带 `MAIN/LAUNCHER`，桌面 launcher 看到两个 entry。

**修复**：`AndroidManifest.xml` 中删除 `MainActivity` 的 `<intent-filter>` 块。`MainActivity` 是图标切换共用的 `targetActivity`，不能自己也是 launcher entry。入口完全交给 `<activity-alias>`。

**注意**：删除后 `flutter run` 会再次报找不到 launch activity。如果需要调试，临时加回 intent-filter 或参考第 3 节的 release manifest 方案。

### 7.4 全部 activity-alias 主题改为 SplashTheme

原来 14 个 `<activity-alias>` 的 `android:theme` 是 `@style/LaunchTheme`（旧方案），改为 `@style/SplashTheme`，让所有桌面入口都走 SplashScreen API。

### 7.5 启动图标全 vector 方案

**根因**：原 `launch_icon_blue.webp`（192×192）放在 `drawable-nodpi/` 时被系统当 1× 处理，拉到 ~768px 显示，严重糊化。`@mipmap/ic_launcher_blue` 在 API 26+ 解析为 adaptive-icon XML，`<bitmap>` 标签无法引用。

**最终方案**：用 Python potrace 把 `plan/图片/PicaComic_original_icons/images/app_icon_no_bg.png`（512×512 透明底）矢量化，生成 `launch_glyph.xml`（7 个 subpath，对应 p+i+c+a 字形 + 3 个字内空），再合并成两个 vector drawable：

**`drawable/launch_icon_full.xml`**（post-splash 阶段用）：
- 两个 `<path>`：浅蓝圆角矩形卡片（`#D4EAFB`，圆角用 `Q` arc 直接画在 path data 里）+ pica 字样（`#2196F3`，evenOdd 填充）
- 圆角直接在 path data 里，不依赖任何容器尺寸，不会变直角

**`drawable-night/launch_icon_full.xml`**（深色模式 post-splash）：
- 同上，卡片色改为 `#2A2930`

**`drawable/launch_splash_icon.xml`**（SplashScreen API 阶段用）：
- 同样的两个 path，但用 `<group scaleX/Y=0.6 pivot=96,96>` 整体缩到中心 60%，四周留 20% 透明边
- 原因：SplashScreen API 会把 `windowSplashScreenAnimatedIcon` 放大填充到固定槽位，外圈约 1/3 被系统裁掉（只保留内 2/3 安全区）。满幅图标会被裁掉圆角和字母边缘，缩进后完整落在安全区内

**`drawable-night/launch_splash_icon.xml`**（深色模式 SplashScreen API）：
- 同上，卡片色改为 `#2A2930`

**4 个 launch_background.xml**（drawable / drawable-v21 / drawable-night / drawable-night-v21）：
- 两层：底色铺满 + `<item width=180dp height=180dp gravity=center drawable=@drawable/launch_icon_full>`

## 8. 动态取色（dynamic 主题）换壁纸回前台 ANR 修复

### 8.1 背景

用户在主题选择里实现了 `dynamic`（Material You 动态取色）挡位：`settings[27] == '0'` 时走 `dynamic_color` 插件的 `DynamicColorPlugin.getCorePalette()`，把系统壁纸取色转成 light/dark ColorScheme 应用到 `MaterialApp`。

实现后出现问题：**换壁纸再切回 App 必现卡死**，弹"picakeep 没有响应"。

### 8.2 根因（已用真机 logcat + 进程状态确认）

- logcat 抓到 MIUI 的 `exp_actresumetimeout` 事件：`ActivityName:LauncherBlue, Duration:10015` —— resume 超时 10 秒，就是 ANR。
- 进程长时间 `R`（满跑 CPU）+ `sticky GC` + 内存飙到 326MB+ —— 典型的反复重建 + 疯狂分配。

链路：换壁纸回前台时 Android（MIUI 尤甚）会**连发一串 `onConfigurationChanged`**（壁纸取色、uiMode 等都算 config 变化）。当时三个触发点全部用 `force: true`，**完全绕过了 `_refreshDynamicColors` 内部那个 600ms 节流**：

1. `MainActivity.onConfigurationChanged` → channel `changed` → `_handleDynamicThemeChanged`
2. `didChangeAppLifecycleState(resumed)`
3. `didChangePlatformBrightness`

每条事件都强制跑一次 `getCorePalette()`（平台通道往返）+ 两次 `corePalette.toColorScheme()`（HCT 色彩量化，纯 Dart 同步、CPU 重）+ 整棵 `MaterialApp` 的 `setState` 重建。几十次全挤在主 isolate 一两帧内 → resume 超时 → ANR。

`force: true` 把本可拦住的节流废掉了，是问题放大的直接原因。

### 8.3 修复（双层防护）

**Dart 侧 `lib/main.dart`**：
- 新增字段 `Timer? _dynamicColorRefreshDebounce`
- 新增方法 `_scheduleDynamicColorRefresh()`：350ms 防抖。风暴期间每来一条事件就 `cancel` 重置定时器、不干活，等系统安定后只刷一次（仍走 `force: true`，但一次风暴只触发一次）
- 三个风暴型触发点（`resumed` / `didChangePlatformBrightness` / `_handleDynamicThemeChanged`）从 `unawaited(_refreshDynamicColors(force: true))` 改为 `_scheduleDynamicColorRefresh()`
- `dispose()` 里 `_dynamicColorRefreshDebounce?.cancel()`，防止卸载后回调
- **保留不变**：启动首次加载 `_scheduleInitialDynamicColorsLoad`（单次，无风暴）继续走直接强制刷新

**原生侧 `MainActivity.kt`**：

### 8.6 后续接手提示

- **dynamic 挡位判定**：`appdata.settings[27] == '0'` 即动态取色，`_buildLightScheme/_buildDarkScheme` 在该挡位且拿到 `_lightDynamicScheme/_darkDynamicScheme` 时优先用动态方案，否则回退 `ColorScheme.fromSeed`。
- **节流/防抖两层不要随意删**：Dart 防抖管"合并成一次"，原生节流管"从源头少发"。只留一层在极端机型（config 事件更密）下可能仍有压力。
- **`toColorScheme()` 是 CPU 热点**：light + dark 两次 HCT 量化都在主 isolate 同步跑。如果以后还要进一步压，可考虑挪到 `compute()`/isolate，但当前防抖后单次开销已可接受，未做此改动。
