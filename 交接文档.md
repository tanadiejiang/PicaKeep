## 6. 后续继续开发时的注意点

### 6.1 不要直接把 `DownloadManager` 硬改成多路径

当前策略是：
- 保留旧的单路径 `DownloadManager`
- 新增独立本地库聚合层处理多来源问题

原因：
- 旧下载删除、封面读取、章节路径、路径迁移都强依赖单路径
- 直接硬改多路径，回归风险很高

### 6.2 图集递归扫描要坚持“叶子目录”原则

否则容易把父目录和子目录同时识别成图集，产生重复项与路径统计错误。

### 6.3 原项目兼容问题优先看运行时读取模型

以后若再遇到：
- 列表能出
- 点详情转圈
- 阅读器打不开
- 某些来源正常、某些来源异常

优先排查顺序建议是：
1. 运行时实际读取了哪个模型
2. `download.db` / 收藏库 / 历史库里的 JSON 结构是否匹配该模型
3. 再看 UI 路由或目录扫描

### 6.4 临时恢复脚本不要当长期业务逻辑依赖

当前脚本适合：
- 一次性修复恢复库
- 对比现有库结构

不适合直接嵌入正式应用逻辑。若后续需要做正式“恢复工具页”，建议单独产品化。
