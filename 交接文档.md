# PicaKeep 交接文档

## 1. 项目定位

`PicaKeep` 当前已经不是单纯的本地离线阅读器裁剪版，而是一个以本地漫画 / 本地图集 / 已下载管理为主线，同时开始落地客户端 / 服务端双模式的 Flutter 项目。

当前工作区中同时存在两个相关工程：
- `D:\Flutter_Projucts\PicaComic\PicaKeep`：当前主要维护目标。
- `D:\Flutter_Projucts\PicaComic\PicaComic`：原项目，对照 UI、阅读器行为、旧数据结构与兼容读取链路时会用到。

当前版本的实际主线已经分成两层：
- **本地层**：本地漫画、本地图集、已下载、收藏、历史、阅读器、设置。
- **远程层**：运行模式、服务信息、局域网扫描、远程资源库 / 远程已下载 / 远程图集、服务端本地 HTTP 与浏览器网页后台。

## 30. 2026-05-28 加密压缩包阅读链与会话密码管理收口

- 本轮收口的问题来自实测：1.7 GB 加密 ZIP（`AAAtest_download (2)`，38 章 / 62 页）输任意非空密码都“解锁成功”，但封面破图、阅读器每页 `Exception: Invalid image data`。
- 根因分三层，互相掩盖：
  1. `archive_reading_service.dart::tryUnlock` 旧实现只调 `backend.openIndex(...)` 并以“能列目录 + 有 image entry”作为成功条件；ZIP 中央目录从来不加密、列目录不需要密码，因此任何密码都被错判为对。
  2. `dart_zip_backend.dart::readEntry` 主路径在解密失败时 `catch (_)` 落到 `_readEntryFallback`，fallback 直接对加密字节做 `RawZLibFilter` inflate，结果是乱码字节流喂给 `dart:ui.ImmutableBuffer.fromUint8List` → `Invalid image data`。
  3. 中央目录解析不识别 WinZip AES extra field `0x9901`，加密 entry 的真实压缩方法 / AES 强度从未被读出来。
- 不能直接换 archive 库解决：实测 archive 3.6.1 的 `ZipFile` 构造硬调 UTF-8 解码 local file header filename，遇到日文 ZIP（`[コデインガール ...]`）抛 `FormatException: Unexpected extension byte`；archive 4.0.9 修了 filename 解码但 `ZipFile.read` 对 zip64 extra field（id=1）的 dataSize 没被 `readBytes` 消费，本仓库的 zip64 包会撞 `RangeError`。两版库都不能透明使用。
- 最终方案：彻底绕开 archive 库的 `ZipDecoder`，自己实现加密 entry 的解密链。
- `lib/foundation/archive/backends/dart_zip_backend.dart`
  - `_parseCentralDirectoryWithOffsets` 在原有 zip64 / extra 循环里追加识别 `0x9901`（vendorVersion / vendorId / encryptionStrength / 真实 compressionMethod），并把 `isAesEncrypted` / `aesStrength` / 内层真实 `compressionMethod` 写回 `_CdEntry`；同时修掉一处 extra field dataSize 越界。
  - `_CdEntry` 增加 `isAesEncrypted` / `compressionMethod` / `aesStrength` 字段；`isEncrypted` 同时考虑 ZipCrypto flag 与 AES extra。
  - `readEntry` 主路径重写：加密 entry 走 `_readAesEntry` 或 `_readZipCryptoEntry`，非加密 entry 走 `_readEntryFallback`，整条主路径不再调用 archive 库的 `ZipDecoder` / `InputFileStream`。
  - `_readAesEntry`：自己读 local file header → 切出 salt / pwdVerify / cipher / file mac → `_pbkdf2HmacSha1` 派生 (encKey, macKey, pwdCheck) → 密码两字节校验失败抛 `wrongPassword` → `pointycastle.HMac(SHA1Digest(),64)` 校验 file mac → `_aesCtrXorZip` 用 `pointycastle.AESEngine` 处理 LE counter（nonce=1 起）逐 block XOR 解密 → 内层 `method=8` 时 `RawZLibFilter` inflate。
  - `_readZipCryptoEntry`：3 个 32-bit 密钥用密码字节初始化（`_zipCryptoInitKeys`），用自实现的 `_zipCryptoCrc32Update`（多项式 0xEDB88320，与 zlib 反射型 CRC32 等价）每字节滚动；前 12 字节 encryption header 中第 12 字节按 ZIP 规范对照 `flags & 0x08 ? lastModFileTime>>8 : crc32>>24`，不一致即 `wrongPassword`；通过则继续解密 payload，再按内层 method 走 store / inflate。
  - `_readEntryFallback` 入口对 `match.isEncrypted` 直接抛 `wrongPassword`，杜绝任何加密字节走裸 inflate 的路径；旧 `catch (_)` 与 `_looksLikeWrongPassword` heuristic 一并删除。
  - `capabilities.supportsAesZip` 改为 `true`，对调用方诚实表态。
- `lib/foundation/archive/archive_reading_service.dart`
  - `tryUnlock` 真实验密码：`openIndex` 后挑一个最小的加密 entry（image 优先）实际 `readEntry` 一次；成功才 `setSessionPassword + evictIndex`；`wrongPassword / passwordRequired` 返回 `false`；其它失败按原码 rethrow 给 UI。
  - 旧的“列目录通过即视为成功”逻辑保留为兜底（应对未加密包仍想触发 unlock 流程的少数边界情况）。

## 32. 2026-05-28 AES-ZIP inflate raw deflate 修复

- 现象：第 30 节收口的 AES-ZIP 解密链在实测 AE-1 + AES-256 + deflate（如 `114514.zip`）时报"压缩包已损坏或格式不正确"，而 AE-2 + ZipCrypto 包（`AAAtest_download (2)`）正常。
- 根因：`_inflateRaw` 调用 `RawZLibFilter.inflateFilter()` 默认 `windowBits=15`，会期待 zlib header（2 字节 CMF/FLG）；而 ZIP entry 的压缩数据是裸 deflate 流（无 header），对应 `windowBits=-15`。Python 验证了同样的明文字节用 `zlib.decompress(data, -15)` 解压完全正确，而默认参数报 `invalid code lengths set`。
- ZipCrypto 包能工作是因为当时也碰巧走了同一路径但密码和内容凑巧让 inflate 不报错，或实测的 entry 是 method=0（store），实际绕过了 inflate。
- 修复：`dart_zip_backend.dart` 中两处 `RawZLibFilter.inflateFilter()` 均改为 `RawZLibFilter.inflateFilter(windowBits: -15)`：
  - `_inflateRaw`（AES / ZipCrypto 解密后调用）
  - `_readEntryFallback` 内的 method=8 分支（非加密 entry 走 fallback 时调用）
- 涉及文件：`lib/foundation/archive/backends/dart_zip_backend.dart`
- 校验：`flutter analyze lib/` 0 issue；Python 同算法验证 inflate 结果正确（size=7832 = usize，开头 `RIFF...WEBPVP8 ` 标准 webp）。

## 33. V1.9.8 Android 图标切换黑条修复与阅读页状态栏动效收口

### 33.1 阅读页顶部黑条（已修复）

**根本原因**

日志中出现 `D/VRI[LauncherBlue]`，说明 launcher 入口是 `LauncherBlue` 这个 `activity-alias`，不是 `MainActivity` 本身。Flutter embedding v2 靠 Activity 上挂的 `<meta-data android:name="io.flutter.embedding.android.NormalTheme">` 在 Flutter UI 起来后切到 NormalTheme。但这条 meta-data 之前只挂在 `<activity>` 上，没有复制到各个 alias。alias 启动时 Flutter 找不到这条，窗口停留在 LaunchTheme，没有 `windowLayoutInDisplayCutoutMode=shortEdges` 也没有 `statusBarColor=transparent`，窗口本身就没把状态栏区域让 app 绘制，所以无论 immersive 怎么调，那块都是黑的。

**修复内容**

`android/app/src/main/AndroidManifest.xml`：给全部 13 个 `activity-alias` 补上：

```xml
android:theme="@style/LaunchTheme"
...
<meta-data
  android:name="io.flutter.embedding.android.NormalTheme"
  android:resource="@style/NormalTheme"/>
```

**注意**

后续如果新增 `activity-alias`，必须同样带上这两项，否则从该 alias 启动时黑条会复现。

### 33.2 阅读页状态栏动效（已修复）

**问题**

隐藏顶栏时，状态栏（系统时间/电量区域）是瞬间消失，而不是像原项目（PicaComic）那样慢慢淡出。

**根本原因**

V1.8 时把移动端 `_hideReaderSystemUi` 从 `SystemUiMode.immersive` 改成了 `SystemUiMode.immersiveSticky`：

- `immersive`：系统栏隐藏时走 Android 标准淡出动画（约 300ms），视觉上"慢慢隐去"。
- `immersiveSticky`：hard-switch，瞬间消失，无淡出动画。

**修复内容**

`lib/pages/reader/comic_reading_page.dart`：`_hideReaderSystemUi` 改回：

```dart
SystemChrome.setEnabledSystemUIMode(SystemUiMode.immersive);
```

**副作用说明**

`immersive` 模式下，用户如果从屏幕顶端下拉，状态栏会出现并停留几秒，需要等系统自动收回。`immersiveSticky` 会在失去交互后立即藏回。这是 V1.8 当时改 sticky 的动机，现在为了动效体验改回 immersive，接受这个副作用。

## 35. 服务端模式（手机当服务端）封面/阅读/重载/幽灵条目系列修复

> 本节记录"手机当服务端、PC 当客户端浏览远程·已下载"链路上的一连串问题修复（内部代号 v1–v7）。背景：服务端浅扫已改为"读 download.db 行 + 不逐图读字节"的轻量模式，深扫只在打开详情/阅读时按需触发。本节所有改动**仅影响服务端响应链路与远程客户端展示**，桌面服务端模式与 Windows 客户端协议保持兼容。

### 35.1 现象与根因（按问题归类）

- **A. 封面大面积破图 / 无法阅读**：`/cover`、详情、`/images` 全走 `_ensureDeepItem → deepScanItem`，而深扫早期 `includeImageSizes:true` 会对**每一页 `_readImageSize`（整页字节经 Shizuku IPC 读出再解析图头）**——解析一个封面 = 把整本所有页字节读一遍。叠加并发闸（安卓仅 2 槽），客户端一次请求 ~40 个封面全部排队，多数在图片超时前拿不到 → 破图、阅读卡死。
- **B. 客户端列表反复转圈重载**：`_computeLibrarySignature` 早期把 `snapshot.generatedAt`（秒级）算进签名 → 每次重扫即使内容没变签名也变 → 发 `library-changed` → 客户端失效重拉。
- **C. 服务端停在"已下载/图集"页，客户端一操作就重载**：`ServerRuntimeState.beginRequest/endRequest` 在**每个 HTTP 请求**进出都 `_notifyChanged → onChanged → App.notifyServiceRuntimeChanged()`，而下载页/图集页把 `serviceRuntimeVersion` 当成"该重载列表"。客户端滚动/翻页产生的请求洪流让宿主页面持续 `reload()`。
- **D. 空 managed 根冒出幽灵条目 "download"**：`_scanManagedRootItems` 在"无 db 记录且目录为空"时走 fallback `_buildManagedFallbackShallowItem(rootPath)`，而该构造器用占位 episode 恒返回非空 item，从不校验目录是否真有图片 → 空根产出一个 0 图、破图、点阅读报"未知错误"的幽灵条目（v2 浅扫重写丢掉了旧深扫 `if(episodes.isEmpty) return null` 的护栏）。
- **E. 页数被硬塞进"作者"字段**：`RemoteLibraryComicItem.subTitle` 在无真实作者时合成 `"X 张图片"/"N 个章节"`，导致卡片作者行、详情页"作者"信息栏都显示成"36 张图片"。
- **F. 详情/底栏大小变"未知"**：打开详情/底栏会 `fetchItemDetail` 触发深扫，深扫改为 `includeTotalBytes:false` 后返回 `totalBytes:0`，覆盖了浅扫从 db `comicSize` 拿到的真实大小 → `comicSize` 变 null → 显示"未知"。

### 35.2 修复内容

- **A 封面/阅读解耦深扫**：新增 `LocalResourceScanner.resolveCoverPathOnly(shallow, rootPath)`——只定位封面文件（先用 db 行的 `coverPath`，`fileExists` 即返回；否则对该本目录列一次找 `cover.*`，再否则取首张可见图），**0~2 次 ls 级 IPC，绝不递归整本、不读整页字节**。`server_app.dart` 封面路由改走它 + 独立的 `_coverPathCache`，不再 `_ensureDeepItem`。深扫 `deepScanItem` 改 `includeTotalBytes:false`，只递归列图片路径供阅读；单页字节由 `/images/{ep}/{page}` 按需经 `_fileResponse` 只读被请求那一页。
- **B 签名去抖**：`_computeLibrarySignature` **删除 `generatedAt`**，签名只由 config 路径 + 各 root 的 `id/path/exists/itemCount/totalBytes` + `totalComicCount/totalBytes` 决定。内容不变 → 签名不变 → `_recomputeLibrarySignature` 的 `changed==false` → 不发 `library-changed`。
- **C 运行信号分流**：新增 `App.serviceStatsVersion` / `App.notifyServiceStatsChanged()`（`app.dart`）。`ServerRuntimeState` 拆成两个回调——`onChanged`（运行/资源变化：start/stop/挡位/库变）与 `onStatsChanged`（请求计数/连接/日志）；`beginRequest/endRequest`/请求日志走 stats。`LocalServerRuntime` 把 `onStatsChanged` 接到 `notifyServiceStatsChanged`（独立防抖）。**列表页（下载页/图集页/me 页）仍只听 `serviceRuntimeVersion`**，因此不再被请求计数洪流触发重载；服务信息页可加听 `serviceStatsVersion` 看实时请求数。
- **D 幽灵条目护栏**：`_buildManagedFallbackShallowItem` 加"目录直接含图片 OR 任一子目录直接含图片"的轻量校验，空目录/空根返回 `null`，不再产出根自身幽灵条目；有 db 记录的正常根 `records` 非空即 return，**完全不进 fallback**，零回归。
- **E 作者字段**：`RemoteLibraryComicItem.subTitle` 改为 `=> subtitle.trim()`——只返回真实作者/副标题，没有就空，**不再合成"X 张图片"/"N 个章节"**。卡片作者行、详情页标题下方与"作者"信息栏空了即收起。（root 条目 `RemoteLibraryRootItem.subTitle` 的"X 部漫画/X 个项目"是另一处合法显示，未改。）
- **F 大小保留**：`deepScanItem` 把 `shallow.totalBytes`（来自 db `comicSize`）作为 `fallbackTotalBytes` 传入 `_scanComicItem`；后者在 `includeTotalBytes:false` 时改用该 fallback，而不是把逐章全 0 的字节求和成 0。深扫只负责枚举图片路径，**不再清空大小**。

## 36. 收藏 / 图片收藏远程档落地 + 远程图片连接池根因修复

> 本节是第 35.5 待办的实现 + 一连串实测排障的最终结论。「收藏」「图片收藏」两页的"远程"档已落地可用；过程中暴露并修复了远程图片加载会**永久毒化连接池**（仅热重启可恢复）的根因。详细过程见临时文档 `交接文档-临时-远程收藏与图片收藏.md`，本节为收口结论。**用户实测："目前来说没有什么问题"。**

### 36.1 远程档功能落地

- 服务端新增收藏 / 图片收藏接口（`server_app.dart` 的 `_handleFavoritesRequest` / `_handleImageFavoritesRequest`），客户端新增模型 `RemoteFavoriteFolder` / `RemoteFavoriteItem` / `RemoteImageFavorite` 与拉取/写入方法（`remote_library_data_source.dart`），两页接 `SegmentedButton` 本地/远程两档。
- 远程档点击进阅读器统一走 `App.pushInner(...)`（**不要用 `App.globalTo`**，会让返回栈跳到「我」页）；`ScaffoldMessenger.of` 一律用 tile/页面自己的 `BuildContext`（**不要用 `App.globalContext`**，根 context 无 Scaffold 后代会断言挂）。
- 图片收藏点击候选 id 用 `image.id` + `downloadId` + `title` + 前缀变体 fallback（`nhentai`/`hitomi`/`jm`/`Ht` 等），因为收藏的 `image.id` ≠ 服务端资源 `remoteId`。
- 两页 `_open`/`_onTap` 都加了 `try/catch + SnackBar`：定位不到弹"该漫画在服务端不可定位"，超时/连不上弹对应提示，**绝不让异常冒泡成 unhandled exception 把当前路由掀掉**（曾导致"点了没反应、弹回我页"）。
- 服务端路径段直接用 `segments[i]`（**不要再 `Uri.decodeComponent`**，`pathSegments` 已自动解码，二次解码会让中文夹炸 500）；favorites handler **不要**每请求 `await LocalFavoritesManager().init()`（会重开 sqlite、阻塞 isolate）。

### 36.2 真·根因：远程图片连接池被永久毒化（最关键）

症状：**滚动经过"服务端实际不存在内容"的漫画/封面后，全部远程档（已下载/图集/收藏/图片收藏）都无法载图，连正常漫画的阅读图也加载不出，离开页面、跨标签都恢复不了，唯有热重启 / 重开 App 才好。**

- **决定性证据**：热重启重建整个 Dart isolate → `RemoteLibraryClient` 单例 + 其 `HttpClient` 连接池清零才恢复 ⇒ 坏的是**单例持有的连接池**，不是服务端、不是 widget。logcat 一片 `getUrl` 的 `TimeoutException 0:00:05`、**无任何 404**。
- **两个泄漏源（均已修）**：
  1. **孤儿连接**：`getUrl()/openUrl().timeout(5s)` 超时只丢弃前端 future，底层开连接的请求仍在跑，稍后成功返回的 `HttpClientRequest` 无人 abort → 永久钉住一条连接，累积到 `maxConnectionsPerHost` 全局假死。
  2. **404 不 drain**：非 2xx（缺失封面）路径原来直接 `request.abort()`，但 `close()` 拿到 response 后 socket 归 response 所有，abort 不可靠 → 每个缺失封面漏一条连接。

### 36.3 连接池修复内容

- **孤儿回收**：新增 `RemoteLibraryClient._openWithTimeout(connecting, timeout)`——超时时给原 `connecting` future 挂 `then`，等它最终返回 request 就 `abort()` 把连接还回池子。`loadImageWithProgress` 的 `getUrl`、`_sendRequest` 的 `openUrl` 都改走它。**新增任何"开连接 + 超时"路径都必须用它，不要再裸 `.timeout()`。**
- **404 改 drain**：非 2xx 路径改为 `await response.drain<void>().timeout(5s)` 读空响应体让连接干净归还；drain 自身超时才回退 `abort()`。
- **并发闸**：`_ImageConcurrencyLimiter`（FIFO 信号量，许可转交式 release，无 double-count）。`loadImageWithProgress` 拿连接前先拿令牌；两个独立预算——**阅读闸** `settings[114]`（reader 大图，默认 6）/ **浏览闸** `settings[115]`（封面缩略图，默认 8）。封面下载传 `isCover:true` 走浏览闸，reader 走阅读闸。令牌释放：早退路径就地 `releasePermit()`，正常路径交给 `_guardedImageBody` 的 `finally`。
- **池扩容**：`_httpClient.maxConnectionsPerHost = 24`（≥ 两闸上限之和 12+12），保证正常并发不在 `getUrl` 排队。**改并发闸上限必须同步抬高这里。**
- **响应体保护**：`_guardedImageBody` 给 body 流套 15s idle timeout，cancel/error/未读完时 `abort()`，正常读完不动（保留 keep-alive）。封面磁盘缓存有失败负缓存（404 冷却 10 分钟、瞬时错误 20 秒）+ `RemoteCoverUnavailableException` 静默占位；`_trimToLimit` 30s 节流且不重入。
- 设置位常量在 `lib/foundation/local_library_settings.dart`：`remoteReaderImageConcurrencySettingIndex=114` / `remoteBrowseImageConcurrencySettingIndex=115`，含 `normalizeRemote*ImageConcurrency`（夹 1~12）；`base.dart` 默认追加 `'6'/'8'`。设置入口在设置-阅读 / 设置-浏览，范围 1~12，实时生效。

### 36.8 硬规则：不要再走的弯路

- **不要**给 `_checkRemoteAvailability` 加缓存"掩盖"问题——它失败的真正原因 99% 是连接池被拖垮，不是探测本身慢。
- **不要**把 `Uri.decodeComponent(segments[i])` 加回去——`pathSegments` 已自动解码。
- **不要**在服务端 favorites handler 里 `await init()`——单例 startup 已 init 过，重复 init 重开 db、阻塞 isolate。
- **不要**用 `App.globalTo` 打开阅读器——永远用 `App.pushInner`，否则返回栈错乱跳「我」页。
- **不要**用 `App.globalContext` 调 `ScaffoldMessenger.of`——用页面/tile 自己的 BuildContext。
- **不要**把图片收藏远程档点击候选缩成只有 `image.id`——要带 `title` + 前缀变体 fallback。
- **不要**在 timeout 路径里只 catch 不 abort——Dart `HttpClient` 的 socket 不会自动释放。
- **不要**只给建连/收 header 包 `.timeout()` 就以为"请求有超时保护"了——**响应体（body）流必须单独保护**：套 idle timeout，且在被取消/出错（未读完）时 `abort()`。裸 `await for(response)` 是这套 codebase 最隐蔽的池泄漏来源。
- **不要**新增 `HttpClient` 实例——应通过 `_httpClient`/`_controlClient` 这两个，否则又开始单池竞争。
- **不要**在"每张封面下载后"无脑跑全盘扫描类清理（`_trimToLimit`）——网格批量加载会把它放大成几十次扫盘，必须节流 + 不重入。
- **不要**对 `getUrl()` / `openUrl()` 直接 `.timeout()` 就完事——**这是孤儿连接的来源**：超时只丢弃前端 future，底层开连接的请求还在跑，稍后成功返回的 `HttpClientRequest` 无人 abort，永久钉住一条连接；累积到 `maxConnectionsPerHost` 就全局假死，**只有热重启/重开 App 能恢复**。必须用 `_openWithTimeout`：超时后给原 future 挂 `then`，等它最终返回 request 时 `abort()` 回收。判断铁证——logcat 全是 `getUrl` 的 5s TimeoutException、没有 404、且热重启后立刻恢复。
- **不要**让 `maxConnectionsPerHost` 小于"所有并发闸上限之和"——否则正常并发就会 oversubscribe → `getUrl` 排队 → 超时 →（若没孤儿回收）雪球。当前 `_httpClient=24` ≥ 阅读闸 12 + 浏览闸 12；改并发闸上限时记得同步检查这里。
- **不要**忘了释放并发令牌——`loadImageWithProgress` 拿令牌后：早退路径（404/超时/socket/其它异常）就地 `releasePermit()`；正常路径把释放责任交给 `_guardedImageBody` 的 `finally`。漏一条路径就会泄漏槽位，最终把闸门耗尽变成假死。

### 36.9 当前可能仍存在的问题

1. **池泄漏是否真的根治了？** 现在两个 client 的所有 timeout/socket 路径都走了 `_openWithTimeout` + drain，理论上池子不该再被废 socket 占。但 Dart `HttpClient` 在某些边缘场景（如 response 已开始流式但 stream 卡住）的释放行为，最好用反复点击 + reader 看 5+ 本后回切场景跑一遍。
2. **首次点击 `findItemByCandidates(fetchDetail: true)` 仍可能慢**。`fetchItems()` 拉全库 + `fetchItemDetail()` 单项详情，两次串行 `_sendRequest`。如果服务端库大，第一次点击可能就 5+8 秒。这种"业务慢"不是 bug，但用户体感差。当前已对第一个匹配项才拉详情、复用 `_detailCache`；真正治本可能要让服务端在 `image-favorites` payload 里直接回传 `itemId`，省掉客户端模糊匹配。
3. **`fetchFavoritesInFolder` 串行循环**。`_loadRemoteFolders` 里逐个文件夹 `await client.fetchFavoritesInFolder(folder.name)`，18 个文件夹就 18 次串行。可改 `Future.wait` 并行（控制并发到 4-6 别打满 `_controlClient`）。当前没改。
4. **reader 退出时的请求清理**。reader dispose 时只 cancel UI 流，**没主动 abort 还在网络上的图片请求**。`abort()` 已能让超时的请求释放 socket，但理论上 reader 主动 abort 所有未完成请求会更干净。可作为下一步增强。

### 36.10 实测脚本

1. 客户端冷启动 → 收藏远程档 → 应直接列出所有文件夹（含中文名）。
2. 图片收藏远程档 → 应显示缩略图，无图项静默破图占位，控制台**不再刷** 404/Timeout。
3. 点击图片收藏远程项 → 进 reader 阅读 → 系统返回一次 → 回到图片收藏页（不是「我」页）。

7. 控制台 grep `[FavRemote]` 应**没有**输出（诊断 print 已清理）。

### 36.11 给下一位 AI 的建议

- 不要被症状牵着走"一个个补丁"。这套问题里 90% 是**连接池/导航栈/编码**这种基础设施级别的根因，每发现一个新症状都先问一句"是不是池子又满了？是不是用错了 navigator？是不是又 decode 了一次？"
- 如果用户报"又有问题"，**先看完整 logcat**，特别是 `flutter` tag 的 unhandled exception 栈，它会精确定位 file:line。
- **保留 `[FavRemote]` 这种诊断 print 模式**作为下次必要时的工具，但**用完要清理**，不要留在生产代码里。
- 任何新增 HTTP 请求路径（不管是控制面还是图片）**timeout/socket catch 必须配 request.abort() 或走 _openWithTimeout**，这是这个 codebase 的硬规定。
- 用户在客户端测试，服务端在另一台手机；服务端日志要单独看（不是客户端 logcat），那里也会有 `ERROR - GET /api/...` 这种关键线索。

### 37.1 ESC 退出不保存阅读进度（只有返回键保存）

- 现象：阅读时按 ESC 退出，阅读到的章节/页码不保存；只有点顶栏返回键才保存。
- 根因：ESC（`onGlobalKeydown`）与返回按钮**同走一个 `back()`**，并非两条分支。阅读翻页/滚动用 `scheduleReaderHistoryWrite()` 挂 **1500ms 防抖**再落盘；退出时 `cleanupReader()` 旧逻辑**直接 `clearTimeout` 清掉待写定时器、不补写**。返回键因唤栏+移鼠标+点击通常已超 1500ms、防抖早落盘，ESC 可在窗口内瞬退被取消。
- 修法：`cleanupReader()` 改调 `flushReaderHistory()`——清定时器后按当前实时页（滚动模式用 `currentScrollPage()` 回写 `pageIndex`）调一次 `writeReaderHistoryNow()`；`lastHistoryKey` 去重保证未移动是 no-op。ESC / 返回键 / 跳转统一经 `cleanupReader`，一处补写全修好。

### 37.2 从历史打开总停在第 1 页（仅滚动连续模式）

- 现象：从历史卡/历史页点进漫画，不从记录页打开而是停在第 1 页。
- 根因：页码全程正确（`openHistory` 传 `pageIndex: record.page` → `renderReader` 存入 → `renderReaderContent` clamp），分页模式直接以该 index 渲染所以正常。滚动连续模式旧实现只有**一次** `requestAnimationFrame(() => scrollToPage(pageIndex))`，`scrollToPage` 用 `scrollIntoView`：目标页之上图片懒加载、跳转那刻高度未稳定→落到顶部；随后 `pageObserver` 按可视页**把 `pageIndex` 覆写**回 0。
- 修法：新增 `restoreScrollPosition(target)`——置 `state.reader.restoring=true`，多帧 rAF 重试 `scrollToPage(target)` 直到 `currentScrollPage()===target` 稳定或超时兜底，`restoreRaf` 在 `cleanupReader` 取消；`pageObserver` 回调加 `if (state.reader.restoring) return;` 守卫，恢复期间不被覆写。`target<=0` 不重试。

### 37.3 历史卡第二行「0」+ 历史页对齐已下载

- 现象：历史卡第二行（作者位）显示「0」；历史页搜索不可用、无卡片/详细切换、卡片样式与已下载不一致。
- 根因：服务端历史记录（`server_app.dart::upsertHistory` / `web_user_store.dart::toJson`）**只存 `target/title/cover/ep/page/maxPage/readEpisode`，没有 author/subtitle/tags**。`readEpisode` 是 int 集合回传 `[0]`，旧 `normalizeHistoryPayload` 用 `String(firstValue(['readEpisode',…]))` 变成 `"0"`，卡片 `item.readEpisode || item.subtitle` 渲染成"0"。
- 修法：① `normalizeHistoryPayload` 不再用 `readEpisode` 当文案；② 作者/副标题与标签改由 `renderHistoryPage` 用 `settle()` 并行拉 `history.list` + `fetchItemsForSource(当前源)`，建 `Map(id→{subtitle,tags})` 补全（资源库缺该条目就留空隐藏）；③ 启用历史页搜索——`updateToolbar` 把 `historyPage` 加进 `searchInput.disabled` 与 `viewSwitchEl.hidden` 白名单，**复用同一 `state.grid.query` / `state.grid.viewMode`（与已下载共用记忆）**，search input/clear/view-switch handler 各加 `historyPage` 分支，过滤用现成 `filterItems`；④ 卡片改用 `.app-card`(`historyPosterCardHtml`)/`.app-detailed-card`(`historyDetailedCardHtml`) 结构，保留右上 `.app-history-remove`。

### 37.4 详细视图超长标题溢出覆盖下一行

### 38.2 收藏项封面破图 + 卡片不可点（核心 bug）

- **根因**：服务端 `_buildFavoriteItemPayload` 原本只回 `name/author/type/tags/target/time/coverUrl`，**没有 `itemId`**；而客户端/网页用 `__matched: !!itemId` 判断是否匹配资源库，于是 `__matched` **永远 false** → 封面被强制清空（`coverItem = {coverUrl:''}`）、卡片标为 `aria-disabled` 不可点。问题**不在** cover 路由本身（`/api/library/favorites/{folder}/{target}/cover` 一直可用），而在 payload 缺字段。
- **修法**：服务端 GET 列文件夹内容（`segments.length == 4`）时先取 `_currentSnapshot()`，对每个收藏项跑 `_resolveFavoriteResourceItem(snapshot, item)`；匹配到就把 `itemId/id/displayId/sourceDisplayName/imageCount/totalBytes/updatedAt` 一并写进 payload，`__matched` 才正确为 true，封面与点击恢复。

### 38.3 收藏封面 fallback（coverPath 失效但资源库有内容）

- cover 路由（`segments.length == 6 && segments[5]=='cover'`）改为三级兜底：
  1. 本地 `item.coverPath` → `_fileResponse`，**要判断返回是否 404**（不能只看路径非空），404 才继续回退；
  2. 命中 `_favoriteCoverFallbackCache`（key = `folder|type|target`）且文件在就用；
  3. 都没有则 `_currentSnapshot()` + `_resolveFavoriteResourceItem` 按 candidates 匹配资源库项，取 `matched.coverPath`，为空再 `_scanner.resolveCoverPathOnly(matched, rootPath: _rootPathForRootId(matched.rootId))`，成功后写入 fallback 缓存。
- `_recomputeLibrarySignature` 里 signature 变化时**连 `_favoriteCoverFallbackCache` 一起 clear**（与 `_deepItemCache`/`_deepItemInFlight` 一起），避免资源移动/删除后命中 stale 封面路径。
- **candidates 复用客户端**：`_buildFavoriteServerCandidates(item)` 直接调 `item.candidateDownloadIds()`（FavoriteItem 自带，含 ehentai/jm/hitomi/Ht/nhentai 前缀变体）再补 `target`；`_findResourceItemByCandidates` 把每个资源项的 `id/title/displayId/sourceTitle/sourceDisplayName/subtitle/path` 收成 pool 做包含匹配。**不要**在服务端另写一套候选规则。
- **cover/删除带 `?type=`**：同一 target 在不同收藏类型下可能并存，`coverUrl` 末尾拼 `?type=${item.type.key}`，`_findFavoriteItemByTarget` 增加可选 `type` 参数精确定位，DELETE 单项同样带 type。
- **`_rootPathForRootId`**：`current_download`/`original_download` 取 config 对应 root，`custom_{n}` 按下标取 `config.customLibraryRoots[n]`，越界返回 null。

### 38.4 详细档卡片对齐客户端 + 标签全显示跟随主题色

- **标题严格 2 行**：`.app-detailed-title` 原本同时有 `text-overflow:ellipsis` 与 `-webkit-line-clamp:2 + white-space:normal`，冲突导致第三行漏出一点。修法：去掉 `text-overflow`，保 `display:-webkit-box; -webkit-box-orient:vertical; -webkit-line-clamp:2; overflow:hidden`，补 `max-height: calc(1.35em * var(--app-detailed-title-lines))`（默认 2）兜底。历史页 `.app-history-entry .app-detailed-title` 仍 3 行覆盖（§37.4），不破坏。
- **标签全显示**：`gridDetailedCardHtml` 去掉 `tags.slice(0,3)`，全量渲染（`.app-detailed-tags` 本就 `flex-wrap:wrap` 自动换行）。只放开**详细档**，海报档/历史档标签数量不动。
- **标签跟随主题色（非写死）**：`.app-detailed-tags span` 背景改 `rgba(var(--accent-rgb),0.12)` + 文字 `var(--accent-2)` + 边框 `rgba(var(--accent-rgb),0.28)`，圆角 8px。主题色由 `:root[data-accent="..."]`（common.css，12 色：蓝/红/粉/紫/靛/青/蓝绿/绿/黄绿/黄/琥珀/橙）切换，用户换主题色时标签随之变色（等效客户端 `secondaryContainer` 主题感）。
